# 执行计划:OpenSpec 分步实施路线图

> 本文档为围标检测系统(DocumentCheck)第一期的 OpenSpec 分步实施路线图,定义 17 个 change 的切分、顺序、验证场景与兜底策略。
>
> **与其他文档的关系**
> - `requirements.md` — 需求全集(功能/非功能/外部依赖)
> - `user-stories.md` — 用户故事与 AC(按 OpenSpec TDD 格式)
> - `task-checklist.md` — 按模块/任务切分的实现清单(9 模块、105 任务)
> - `execution-plan.md`(本文) — 跨模块的 **change 级** 推进路线

---

## 1. 概述

- **粒度方案**:方案 B(按 capability 切,中粒度),Detection 细化为每个 Agent 一个 change,元数据 3 个 Agent 合并为一个 change(同数据源+同算法骨架)
- **change 总数**:17
- **推进方式**:严格串行,一人推进,每个 change 归档后再开下一个
- **里程碑**:M1 基础就绪 → M2 核心流程可跑 → M3 检测可执行 → M4 可交付

---

## 2. 切分原则

### 2.1 粒度约定

- 每个 change 对应一个**用户可感知的 capability**,不是一个技术任务、也不是一整个阶段
- 每个 change 包含 **3-8 个端到端验证场景**,超过 10 个考虑拆分
- 每个 change 独立可 proposal、可 apply、可 archive,且归档后下一个 change 才开始

### 2.2 推进约定

- 严格 C1 → C17 顺序,不并行
- 每个 change 实施流程:`openspec-propose` → `openspec-apply` → 跑验证场景 → `openspec-archive`
- 每个 M 里程碑结束时记录一段**演示级操作凭证**(截图或简要说明)

### 2.3 过程兜底(贯穿 17 个 change)

1. 任一 change 实施中发现依赖未就绪 → **暂停**该 change、回补依赖,不临时 mock
2. 任一 change 范围失控(scenario > 10) → **就地二次拆分**,不硬塞
3. 每个 change 归档前必须跑完该 change 的全部验证场景,**全绿才 archive**
4. 每个 M 结束必须有**演示级交付凭证**,无凭证不进入下一 M

### 3.2 各 Change 详表

#### C1 `infra-base`

- **对应范围**:TS-1~6(DB/异步/SSE/LLM 适配/前端基础/数据生命周期)
- **核心能力**:DB 连接+Alembic 迁移框架、SSE 推送基础、LLM 适配层(统一接口+配置注入)、数据生命周期清理定时任务
  > **范围调整(2026-04-14)**:异步任务框架移至 C6。理由:C1 阶段无真实消费者(Parser/Detection 都未实现),独立验证困难,避免提前做宽。
- **验证场景**
  1. 执行 `alembic upgrade head`,所有基础表建起来
  2. 启动后 `/health` 返回 200 且能连上 DB
  3. `/demo/sse` 端点可连接,前端能收到心跳
  4. LLM 适配层单测:mock provider 通过统一接口调用,超时/限流走降级分支
  5. 生命周期任务 dry-run:能标记过期文件但不删,输出清单
- **兜底**
  - LLM provider 不可用 → 返回结构化 error,由上层决定降级策略
  - DB 断连 → 自动重连
  - SSE 客户端断开 → 客户端自动重连、服务端幂等

---

#### C2 `auth`

- **对应范围**:US-1.1 ~ US-1.4
- **核心能力**:登录(JWT+失败计数+账户锁定)、登出、路由守卫+角色鉴权、初始用户 seed 与改密
- **验证场景**
  1. 首次部署 seed 默认账号 → 登录成功拿到 JWT
  2. 错密 5 次 → 账户锁定,第 6 次提示锁定
  3. 过期 token 访问 API → 401
  4. 普通用户访问 `/admin/*` → 403
  5. 改密后旧 token 立即失效
- **兜底**
  - 账号锁定设 TTL,超时自动解锁
  - seed 幂等(多次启动不重复创建)
  - JWT 黑名单用短 TTL,过期自然清理

---

#### C3 `project-mgmt`

- **对应范围**:US-2.1 ~ US-2.4
- **核心能力**:项目创建/列表(分页+筛选+搜索)/详情/删除
- **验证场景**
  1. 创建项目 → 列表出现
  2. 分页 + 按状态筛选 + 关键字搜索
  3. 详情页包含投标人数/报告数/进度占位
  4. 软删除 → 列表消失,DB 记录保留(审计)
- **兜底**
  - 并发创建同名项目幂等
  - 删除前校验是否有进行中的检测任务

---

#### C4 `file-upload`

- **对应范围**:US-3.1 ~ US-3.4 + US-4.1 + US-4.4
- **核心能力**:投标人 CRUD、压缩包上传(zip bomb 防护+路径穿越防护)、文件列表/删除、加密包解密、报价规则配置(币种/含税/单位)
- **验证场景**
  1. 上传正常 zip/7z → 投标人条目+文件清单出现
  2. 上传含 `../etc/passwd` 的包 → 拒绝并记录告警
  3. 上传 zip bomb(压缩比超阈值)→ 拒绝
  4. 上传加密包 → 提示输密码 → 二次提交成功
  5. 配置报价规则 → 保存并回显
- **兜底**
  - 上传中断可续传
  - 解压失败保留原包供人工处理
  - 密码错 3 次冻结该文件

---

#### C5 `parser-pipeline`

- **对应范围**:L-1(角色分类 LLM)、L-2(报价识别 LLM)、US-4.2、US-4.3
- **核心能力**:DOCX/XLSX 文本提取、LLM 角色分类、报价识别、人工角色修正
- **验证场景**
  1. 含"投标函/商务/技术/报价表"的标准包 → 自动分类正确率目测 ≥ 80%
  2. LLM 返回格式异常 → 降级为关键字规则分类
  3. 人工修正某文件角色 → 下游依赖被标为待重算
  4. XLSX 合并单元格/千分位/大写金额 → 能抽到数字
  5. 解析失败文件在 UI 有"解析失败"标记+重试按钮
- **兜底**
  - LLM 不可用走规则兜底
  - 单文件解析失败不阻塞整包
  - 解析结果可人工覆盖

---

#### C6 `detect-framework`

- **对应范围**:US-5.1 ~ US-5.4
- **核心能力**:启动检测 API、**异步任务框架**(asyncio + ProcessPoolExecutor,从 C1 迁入)、Agent 并行调度(asyncio gather + 超时 + 重试)、SSE 进度推送、综合研判骨架
- **验证场景**
  1. 用 2 个 dummy Agent 跑完整链路 → SSE 看到启动/进行中/完成
  2. 模拟一个 Agent 抛异常 → 整体不崩,该维度标"失败",其他继续
  3. 模拟超时 → 触发兜底降级,整体继续
  4. 重复启动同项目检测 → 幂等(返回同一 task 或拒绝)
  5. 综合研判骨架(占位 regex)能把各 Agent 结果聚合展示
- **兜底**
  - 单 Agent 失败隔离,不影响其他
  - 超时强制回收进程
  - 中途取消可清理现场

---

#### C7 `detect-agent-text-similarity`

- **核心能力**:整文档级文本相似度 Agent
- **验证场景**
  1. 2 份人为抄袭样本 → 高相似度命中
  2. 2 份独立样本 → 低相似度不误报
  3. 3 份中 1 对抄袭 → 只标那一对
  4. 极短文档(< 阈值)→ 跳过并说明原因
  5. LLM 不可用 → 退化为本地向量算法
- **兜底**:LLM 失败 → 本地向量算法兜底

---

#### C8 `detect-agent-section-similarity`

- **核心能力**:章节级相似度 Agent(按标题切章对比)
- **验证场景**
  1. 两份文档在"技术方案"章节雷同 → 命中该章节
  2. 两份文档整体相似但章节切分不同 → 按规范化后对齐比较
  3. 章节识别失败的文档 → 标"章节切分失败",不假阳
  4. 极少章节文档 → 跳过
- **兜底**:章节切分失败 → 降级为整文档粒度(与 C7 结果合并去重)

---

#### C9 `detect-agent-structure-similarity`

- **核心能力**:结构相似度 Agent(目录结构/字段结构/表单填充模式)
- **验证场景**
  1. 两份投标书目录完全一致(含错别字)→ 命中
  2. 两份报价表填充结构一致(相同空值/相同合并单元格)→ 命中
  3. 结构差异明显的独立文档 → 不误报
  4. 结构提取失败 → 标"结构缺失"
- **兜底**:结构提取失败 → 跳过该维度,不假阳

---

#### C10 `detect-agents-metadata` (合并 3 个 Agent)

- **核心能力**:作者/修改时间/机器指纹 三个子 Agent 共用同一元数据提取器
- **合并理由**
  - 共用数据源(`core.xml` / `app.xml` / `docProps`)
  - 同算法骨架(跨投标人的字段聚类/碰撞)
  - UI 呈现耦合(合并展示为"元数据维度表")
- **验证场景**
  1. 2 份 DOCX 的 `core.xml` author 相同 → 命中作者相同
  2. 连续在 5 分钟内被修改 → 命中时间聚集
  3. `appVersion` + `template` 碰撞 → 命中机器指纹
  4. 元数据被清洗 → 标"元数据缺失",不假阳
  5. 子检测可通过配置项(flag)单独关闭
- **兜底**
  - 元数据缺失 → 标"数据不足",不假阳
  - 提取器失败 → 整 Agent 标"失败",不影响其他维度

---

#### C11 `detect-agent-price-consistency`

- **核心能力**:报价雷同 Agent(尾数/金额模式/报价表项)
- **验证场景**
  1. 3 家报价尾数完全一致 → 命中
  2. 2 家报价项明细 95% 以上相同 → 命中
  3. 币种/含税口径不一致 → 先归一化再比较
  4. 异常样本(非数值/缺失)→ 跳过不假阳
- **兜底**:归一化失败 → 标"口径不一致,无法比对"

---

#### C12 `detect-agent-price-anomaly`

- **核心能力**:异常低价 Agent(相对均值/相对标底)
- **验证场景**
  1. 某家明显低于均值 N% → 命中
  2. 含税口径已归一 → 按统一口径判断
  3. 样本 < 3 家 → 保守不触发
  4. 标底未配置 → 仅按均值判断并说明
- **兜底**:样本不足 → 跳过,报告中说明

---

#### C13 `detect-agent-bidder-relation`

- **核心能力**:项目内投标人关联 Agent(电话/地址/法人/联系人碰撞)
- **验证场景**
  1. 两家共用联系电话 → 命中
  2. 两家地址高度相似(规范化后同街同号)→ 命中
  3. 法人/联系人同名 → 命中
  4. 关联字段缺失 → 标"数据不足"
- **兜底**:字段缺失 → 跳过该子项,不假阳

---

#### C14 `detect-agent-history-cooccur`

- **核心能力**:跨项目历史共现 Agent
- **验证场景**
  1. 两家在历史多个项目中共同中标/陪标 → 命中
  2. 历史数据库为空 → 标"历史数据不足",不假阳
  3. 仅一次共现 → 不触发
- **兜底**:历史库缺失 → 整 Agent 跳过

---

#### C15 `report-export`

- **对应范围**:US-6.1 ~ US-6.6
- **核心能力**:报告总览/维度明细/对比入口/检测日志/人工复核/Word 导出
- **验证场景**
  1. 完成检测 → 报告总览渲染 6 张卡
  2. 进入维度明细看证据+原文引用
  3. 人工复核改判 → 报告实时更新+记录操作者
  4. 导出 Word → 含封面/结论/10 维度/证据截图
  5. 日志按 Agent/状态/时间过滤
- **兜底**
  - Word 模板加载失败 → 降级为纯文本导出
  - 证据截图失败 → 占位图+说明

---

#### C16 `compare-view`

- **对应范围**:US-7.1 ~ US-7.3
- **核心能力**:文本对比/报价对比/元数据对比三类视图
- **验证场景**
  1. 选 2 家投标人 → 文本 diff 高亮
  2. 报价对比表按项展开
  3. 元数据对比表格+高亮异同
  4. 超大文件(> 50MB)对比 → 分页加载
- **兜底**:大文件 → 分页加载+只对比前 N 章节

---

#### C17 `admin-users`

- **对应范围**:US-8.1 ~ US-8.4
- **核心能力**:用户列表/创建/启用禁用、规则配置(报价/权重/阈值)
- **验证场景**
  1. 创建用户 → 立即可登录
  2. 禁用用户 → 现有 session 立即失效
  3. 修改规则 → 下次检测立即生效
  4. 规则有版本号,可回滚到上一版本
- **兜底**:规则版本号便于回滚
