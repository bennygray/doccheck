> 版本: v0.8  
> 日期: 2026-04-13  
> 状态: 草稿（经 5W1H + 第一性原理推导细化 + 六轮 review）

---

## 1. 项目背景与目标

### 1.1 背景

在招投标过程中，围标/串标是常见的违规行为。投标人之间通过串通投标价格、共用投标文件、同一主体操控多家投标等方式，破坏招投标市场的公平竞争秩序。目前对围标行为的识别主要依赖人工审查，效率低、主观性强、容易遗漏。

### 1.2 目标

建设一套**基于投标文件多维分析的围标检测系统**，通过自动化解析投标文件，结合 LLM 大模型的语义理解能力，从文本内容、文档元数据、报价数据、图片资源等多个维度进行交叉比对，识别围标/串标风险，输出包含 LLM 综合研判结论的检测报告。

### 1.3 核心价值

- 将人工数天的审查工作压缩为系统自动化分析
- 覆盖人工难以发现的隐蔽围标特征（如硬件指纹、图片哈希、元数据、语言风格）
- LLM 提供语义级判断能力，超越传统关键词/统计方法
- 提供量化评分、证据链和 LLM 生成的自然语言结论，辅助决策

### 1.4 系统定位

| 项 | 结论 |
|---|------|
| 使用阶段 | 评标前快速筛查 + 评标后深度审计，两种模式 |
| 文件来源 | SRM 系统导出 + 手动上传混合，第一期以手动上传为主 |
| 报告定位 | 内部风控参考，辅助决策，允许适度误报，重在发现线索 |
| LLM 角色 | 文档理解、语义判断、结论生成（第二期: 对话交互） |

## 2. 用户角色与权限

### 2.1 角色定义

| 角色 | 描述 | 权限范围 |
|------|------|---------|
| 管理员 | 系统管理人员 | 查看所有项目、用户管理、检测规则/阈值配置 |
| 审查员 | 负责日常招标文件审查的业务人员 | 创建项目、上传文件、发起检测、查看报告、人工复核 |

### 2.2 权限规则

- 系统需要登录界面，所有操作需认证
- 审查员之间数据隔离：只能看到自己创建的项目
- 管理员可查看所有项目（用于工作交接等场景），对他人项目为**只读**：可查看详情、查看报告、导出 Word，但不可执行写操作（添加/删除投标人、上传文件、启动检测、提交复核）
- 第一期采用严格独占模式：项目只有创建者和管理员可见，不支持指派协作

### 2.3 初始用户

- 系统部署后，数据库初始化脚本自动创建默认管理员账号（admin / admin123）
- 默认管理员首次登录后强制修改密码，修改前不允许执行其他操作
- 后续用户由管理员在"用户管理"页面创建

## 3. 业务流程

```
创建招标项目 → 上传投标文件包(≥2家) → 系统自动解析 → 启动检测分析 → 渐进式出结果 → 生成检测报告 → 人工复核确认
```

### 3.1 详细流程

1. **创建招标项目**: 审查员填写项目基本信息（项目名称、招标编号、最高限价/控制价等）
2. **上传投标文件**: 按投标人分别上传应标文件压缩包，每个压缩包对应一个投标人，上传时指定投标人名称
3. **系统自动解析**: 系统自动解压、识别文件类型、分类文档角色、提取文本/元数据/报价/图片
4. **启动检测分析**: 至少上传 2 个投标人后即可启动。用户只需点"开始检测"，各维度 Agent 自动判断是否需要执行，并行运行，渐进式展示结果
5. **渐进式出结果**: 每个 Agent 完成后立即展示该维度的结果摘要，用户无需等全部完成
6. **生成检测报告**: 所有 Agent 完成后，LLM 综合研判，输出完整的风险结论报告
7. **人工复核**: 审查员对系统判定结果进行确认或调整，给出最终结论
8. **重新检测（可选）**: 追加投标人后或调整规则阈值后，可再次点击"开始检测"。第一期执行全量重跑，第二期支持增量检测（仅跑新增配对）

### 3.2 状态机

#### Project 状态转换

```
draft ──[上传文件触发解析]──→ parsing
parsing ──[所有解析完成 且 ≥2投标人completed]──→ ready
parsing ──[所有解析完成 但 <2投标人completed]──→ draft
ready ──[新增文件触发解析]──→ parsing
ready ──[点击"开始检测"]──→ analyzing
ready ──[删除投标人 且 剩余completed投标人<2]──→ draft
analyzing ──[报告生成完成]──→ completed
completed ──[新增文件触发解析]──→ parsing
completed ──[点击"开始检测"重跑]──→ analyzing
completed ──[删除投标人 且 剩余completed投标人≥2]──→ ready
completed ──[删除投标人 且 剩余completed投标人<2]──→ draft
```

- **analyzing 期间锁定**: 不可添加/删除投标人和文件，不可删除项目
- **completed 删除投标人**: 已有分析结果失效，项目回退到 ready 或 draft（取决于剩余投标人数量），前端提示"投标人已变更，当前报告基于变更前数据，建议重新检测"
- **ready 判定条件**: 无活跃解析的投标人 + ≥2 个 parse_status=completed 的投标人
- **completed → analyzing**: 允许不追加投标人直接重跑（如管理员修改了规则阈值后重新检测）

#### Bidder 状态转换

```
pending ──[文件开始解压解析]──→ parsing
parsing ──[所有文件处理完毕 且 ≥1个文件成功]──→ completed
parsing ──[所有文件处理完毕 且 全部失败/跳过]──→ failed
completed ──[追加上传新文件]──→ parsing
failed ──[重新上传/输入密码重试]──→ parsing
```

- **completed** 不要求所有文件都成功——有成功的即算 completed，个别文件失败不阻塞整体流程
- **failed** 仅当该投标人下没有任何成功解析的文件

#### BidDocument 状态转换

```
pending ──[开始解析]──→ parsing
parsing ──[解析成功]──→ completed
parsing ──[解析出错]──→ failed
pending ──[格式不支持]──→ skipped
```

- **skipped**: 格式不支持（DOC/XLS/PDF 等），主动不处理，不计入"解析失败"统计
- **failed**: 尝试处理但出错（文件损坏、超时等），计入失败统计并展示错误原因

#### 进程重启恢复

系统启动时扫描并修复异常状态:
- BidDocument 处于 parsing → 重置为 failed（parse_error="进程重启，解析中断"）
- Bidder 处于 parsing → 根据其文件状态重新计算（若有 completed 文件则 completed，否则 failed）
- Project 处于 analyzing → 重置为 ready，未完成的 AgentTask 标记为 failed
- 前端下次访问时用户看到状态已恢复，可手动重新触发

## 4. 功能需求

### 4.1 项目管理

#### F-PM-01 创建招标项目

- 填写: 项目名称、招标编号、最高限价/控制价（选填）、项目描述（选填）
- 创建后状态为"草稿(draft)"

#### F-PM-02 项目列表

- 展示当前用户可见的项目（审查员仅自己的，管理员全部）
- 支持按状态、时间、风险等级筛选和排序
- 列表展示: 项目名称、投标人数量、检测状态、风险等级、创建时间

#### F-PM-03 项目详情

- 查看项目基本信息
- 查看该项目下所有投标人及其文件列表
- 查看解析状态和检测状态

#### F-PM-04 删除项目

- 删除项目及其关联的所有投标人、文件、分析结果
- 需二次确认

### 4.2 文件上传与管理

#### F-FM-01 上传投标文件包

- 每个投标人单独上传一个压缩包，上传时指定投标人名称
- 支持拖拽上传和点击上传
- 支持的压缩格式: ZIP、RAR、7Z
- 支持同一投标人追加上传（多个压缩包合并）
- 单个压缩包大小限制: 500MB
- 上传后自动触发解压和解析流程

#### F-FM-02 解压处理

- 递归解压嵌套压缩包（最大嵌套深度: 3层）
- 支持 ZIP/RAR/7Z 格式
- 处理加密压缩包: 标记为"需密码"，提示用户输入密码后重试
- 防 zip bomb: 限制解压后总文件大小不超过 2GB、文件数量不超过 1000
- **防 Zip Slip 目录穿越**: 解压每个文件前校验其路径，拒绝包含 `..` 或绝对路径的条目，确保所有文件解压到目标目录内部。使用 `os.path.realpath()` 验证最终路径是否在预期目录下，不合法条目跳过并记录警告
- 处理文件名编码: 自动尝试 UTF-8 / GBK / GB2312 解码

#### F-FM-03 文件类型识别

系统需支持以下文件类型的识别和处理:

| 类别 | 文件格式 | 处理方式 |
|------|---------|---------|
| 文本文档 | DOCX | 直接解析 |
| 报价表 | XLSX | 直接解析 |
| 图片 | JPG / PNG / BMP / TIFF | 记录哈希值 |
| 不支持 | DOC / XLS / WPS / RTF / PDF 等 | 标记为"格式不支持，请转换为 DOCX/XLSX 后重新上传" |
| 其他 | 未知格式 | 标记跳过，不影响整体流程 |

注: 第一期仅支持 DOCX 和 XLSX 现代格式，不支持 DOC/XLS 旧格式、PDF 和 OCR。

#### F-FM-04 文档角色分类

系统自动识别每个文件的业务角色，作为后续交叉比对的对齐依据。

分类策略：规则预分类（文件名关键词匹配） + LLM 批量确认/修正。LLM 按投标人批量处理，一次调用接收一个投标人的全部文件名 + 每个文件的首段文本，同时完成两项任务:
1. 输出所有文件的角色分类
2. 提取该投标人的标识信息（公司全称、简称、关键人员姓名、资质编号等），用于后续错误一致性交叉检索

文档角色定义:

| 角色 | 识别依据 |
|------|---------|
| 技术方案 | 技术方案、技术标、技术建议书 |
| 施工组织设计 | 施工组织、施工方案、施工设计 |
| 报价清单 | 报价、清单、工程量、商务标、投标报价 |
| 综合单价分析表 | 综合单价、单价分析 |
| 投标函 | 投标函、投标书 |
| 资质证明 | 资质、资格、营业执照 |
| 企业介绍 | 企业介绍、公司简介、公司概况 |
| 授权委托书 | 授权、委托 |
| 其他 | 以上均不匹配 |

- LLM 分类置信度低的文件标记为"待确认"并在前端高亮提示，引导审查员主动审阅
- 支持审查员手动修正自动分类结果
- 角色修正后不改变项目状态；若项目已完成检测(completed)，前端展示提示 banner 建议重新检测

#### F-FM-05 文件列表与状态

- 按投标人分组展示文件列表
- 每个文件展示: 文件名、文件类型、文档角色、文件大小、解析状态
- 解析状态: 待解析 / 解析中 / 已完成 / 失败
- 失败时展示失败原因

### 4.3 文档解析

#### F-PA-01 文本提取

- 从 DOCX 文件中提取全文文本
- 按段落分段存储，保留段落结构
- 文本框（text box）内容提取，归入所在位置的文本流中
- 页眉页脚单独提取，用于错误一致性检测（公司名混淆），不参与文本相似度计算
- 仅支持 DOCX 格式，旧格式（DOC/WPS/RTF）需用户自行转换后上传

#### F-PA-02 元数据提取

从文档属性中提取以下元数据字段:

| 字段 | 来源 | 用途 |
|------|------|------|
| 作者 (author) | 文档属性 | 软件特征比对 |
| 最后保存者 (last_saved_by) | 文档属性 | 软件特征比对 |
| 公司名称 (company) | 文档属性 | 软件特征比对 |
| 创建软件 (creator/producer) | 文档属性 | 软件特征比对 |
| 创建时间 (created_at) | 文档属性 | 操作时间分析 |
| 修改时间 (modified_at) | 文档属性 | 操作时间分析 |
| 计价软件锁号 | 特定报价软件输出 | 软件特征比对 |
| MAC地址 / CPU序列号 / 硬盘序列号 | 文档嵌入信息 (如有) | 硬件特征比对 |
| raw_metadata | 原始元数据全量 (JSON) | 备查 |

硬件信息提取说明: MAC/CPU/硬盘序列号通常不在标准文档元数据中，仅在特定编标/计价软件（如广联达、品茗等）生成的文件中存在。需针对常见软件输出格式做专项适配，必要时 LLM 辅助从非结构化文本中识别硬件信息片段。

元数据过滤: 通用值（如 Administrator、User、默认用户名）不算有效匹配，需维护过滤白名单。

#### F-PA-03 报价数据提取

- 从报价类 Excel/Word 文件中解析工程量清单
- 提取字段: 项目编码、项目名称、单位、数量、单价、合价
- 提取综合单价分析明细 (如有)
- 计算投标总价

报价表解析策略（因格式不统一）:
1. 程序提取 Excel 原始单元格数据
2. LLM 理解表结构，识别列映射（哪列是编码、哪列是单价等）
3. 同一项目内，LLM 只需理解一次格式（招标方通常提供统一模板），输出解析规则后程序批量应用到该项目所有投标人
4. 系统展示识别结果供用户确认/修正

#### F-PA-04 图片提取

- 从 DOCX 文件中提取所有嵌入图片
- 计算每张图片的 MD5 哈希值（精确匹配）
- 计算每张图片的 pHash 感知哈希值（模糊匹配，防微调逃避）
- 记录图片在文档中的位置

#### F-PA-05 解析流水线

用户上传一个投标人的压缩包后，系统按以下步骤异步处理。每个投标人的流水线是独立的 asyncio task，多个投标人并行互不阻塞。

**API 同步处理（立即返回 201）:**
- 保存压缩包到磁盘
- 创建 Bidder 记录（parse_status=parsing）
- 更新 Project.status → parsing
- 启动后台异步任务

**步骤1: 解压（进程池）**
- 递归解压，执行安全校验（总大小≤2GB、文件数≤1000、嵌套≤3层）
- 为每个解压出的文件创建 BidDocument 记录（status=pending）
- 格式不支持的文件直接标记 status=skipped
- 推送 SSE: parse_progress（解压完成，N个文件）

**步骤2: 逐文件提取（进程池，各文件并行）**
- DOCX: 文本提取（段落+文本框+页眉页脚）+ 元数据提取 + 图片提取（MD5+pHash）
- XLSX: 单元格数据提取 + 元数据提取 + 隐藏 sheet/列扫描（硬件信息）
- 独立图片文件: MD5 + pHash 计算
- 每个文件完成后立即更新 BidDocument.status=completed 或 failed
- 推送 SSE: parse_progress（提取中，M/N完成）

**步骤3: LLM 角色分类 + 标识提取（asyncio协程）**
- 依赖：步骤2全部完成（需要所有文件的首段文本）
- 按投标人 1 次 LLM 调用（见 L-1 规格）
- 更新 BidDocument.doc_role + Bidder.identity_info
- 兜底：LLM 失败时用关键词匹配角色，标识信息留空
- 推送 SSE: parse_progress（角色分类完成）

**步骤4: 报价表结构识别 + 报价提取（asyncio协程，条件触发）**
- 触发条件：当前投标人有报价类文件 且 该项目尚无 PriceParsingRule
- 对该项目的第一个报价文件调用 LLM 识别结构（见 L-2 规格）
- 识别成功后：
  - 生成 PriceParsingRule 记录
  - 用规则提取当前投标人的报价数据 → PriceItem
  - **自动回填**：扫描该项目下所有已解析完成但尚未提取报价的投标人，用同一规则批量提取
- 并发控制：同一项目多个投标人同时到达步骤4时，仅第一个执行 LLM 识别，后续等待规则生成后直接应用
- 已有 PriceParsingRule 时：直接应用现有规则提取当前投标人的报价数据
- 兜底：识别失败 → 标记待用户手动配置列映射
- 推送 SSE: parse_progress（报价解析完成）

**步骤5: 收尾**
- 更新 Bidder.parse_status → completed（≥1个文件成功）或 failed（全部失败/跳过）
- 级联检查 Project 状态：所有 Bidder 解析完毕时，≥2个 completed → ready，否则 → draft
- 推送 SSE: parse_progress（该投标人解析完成）

### 4.4 检测分析

#### 4.4.1 架构：Agent 并行检测 + 渐进式出结果

##### 核心设计理念

- **"程序做捞网、LLM 做鉴定"**: 程序负责确定性计算（哈希、数值、统计），LLM 负责需要理解和判断的环节
- **每个维度是独立 Agent**: 自主判断是否需要执行，自主处理兜底逻辑，并行运行互不阻塞
- **用户零配置**: 用户只需点"开始检测"，无需选择维度，Agent 自动判断
- **所有环节必须有兜底方案**: 每个 Agent 内部处理失败/降级/跳过逻辑，对用户无感

##### Agent 自判断机制

每个维度 Agent 启动时先检查前置条件，决定是否执行:

| Agent | 前置条件检查 | 不满足时的行为 |
|-------|-------------|---------------|
| 硬件特征码 Agent | 是否提取到硬件信息 | 跳过，报告"未检测到计价软件输出，此维度不适用" |
| 价格构成 Agent | 是否存在报价类文件 | 跳过，报告"未发现报价清单文件" |
| 报价规律 Agent | 是否有 ≥3 家投标人的报价数据 | 跳过，报告"投标人数量不足，无法进行规律分析" |
| 接近限价 Agent | 项目是否填写了最高限价 | 跳过，报告"未设置最高限价，此维度不适用" |
| 错误一致性 Agent | 是否提取到投标人标识信息 | **不跳过，降级运行**: identity_info 为空时退化为用投标人名称做纯关键词交叉搜索，降级模式下不做铁证判定，标记"降级检测" |
| 其他 Agent | 是否有对应角色的文档 | 跳过，报告具体原因 |

##### 流水线与 Agent 依赖关系

系统不是严格的"三阶段串行"，而是**各 Agent 各自的前置条件满足即启动**，最大化并行度。

```
解压 + 文本提取 + 元数据提取 + 图片提取+哈希（程序，各投标人并行）
    │
    ├── 元数据就绪 ──→ 软件特征 Agent（无需等角色分类）
    │                  操作时间 Agent（无需等角色分类）
    │                  硬件特征 Agent（无需等角色分类）
    │
    ├── 文档角色分类（LLM，同时提取投标人标识信息）
    │       │
    │       ├── 角色对齐完成 ──→ 文本相似度 Agent
    │       │                    错误一致性 Agent
    │       │                    语言风格 Agent
    │       │                    图片复用 Agent
    │       │
    │       └── 报价表结构识别（LLM，每项目 1 次）
    │               │
    │               └── 报价数据提取完成 ──→ 价格构成 Agent
    │                                       报价规律 Agent
    │                                       接近限价 Agent
    │
    └── 全部 Agent 完成或超时 ──→ LLM 综合研判 → 生成结论报告
```

关键依赖说明:
- **不依赖角色分类的 Agent**（可更早启动）: 软件特征、操作时间、硬件特征 — 只需元数据
- **依赖角色分类的 Agent**: 文本相似度、错误一致性、语言风格、图片复用 — 需要同角色文档对齐
- **依赖报价数据的 Agent**: 价格构成、报价规律、接近限价 — 需要报价表解析完成
- **综合研判的触发**: 全部 Agent 完成或超时后触发（某 Agent 卡住不阻塞最终报告生成）

##### 综合研判 LLM 输入策略

综合结论应站在项目整体视角撰写，不逐对罗列。LLM 输入不是原始 Pair 数据，而是**程序预聚合的摘要**：

1. **铁证级发现**：命中的维度 + 涉及的投标人名称 + 关键证据一句话
2. **高风险投标人分组**：程序根据 Pair 评分将投标人聚类为疑似围标群组（如 A-B-C 三方互相高分），而非逐对列举
3. **各维度触发概览**：每个维度一行（触发/未触发/跳过 + 最突出发现的一句话摘要）
4. **整体分析结果**：报价规律、限价接近、时间异常的结论
5. **项目基本信息**：投标人数量、项目名称等上下文

LLM 输出要求：综合风险判断结论（2-3 段自然语言），重点回答"是否存在围标嫌疑、主要证据是什么、涉及哪些投标人"。

前后端通信: 后端通过 **SSE（Server-Sent Events）** 实时推送状态变化，前端 EventSource 监听。使用 FastAPI 内置 `EventSourceResponse`（v0.135+），最简实现，不做自定义心跳管理和断线补发。浏览器 EventSource 自带自动重连；重连后前端通过 API 拉取最新状态恢复。

SSE 事件类型及 payload:

**`parse_progress`** — 投标人解析进度
```json
{
  "bidder_id": "uuid",
  "bidder_name": "A公司",
  "step": "decompressing | extracting | classifying | pricing | completed | failed",
  "progress": {"done": 5, "total": 12},
  "error": "..."
}
```
- `progress` 仅 extracting 步骤有值（已完成/总文件数）
- `error` 仅 failed 时有值

**`agent_status`** — Agent 状态变更
```json
{
  "agent_name": "text_similarity",
  "status": "waiting | running | completed | skipped | failed | timeout",
  "score": 75.5,
  "summary": "发现3组高相似段落",
  "skip_reason": "...",
  "fail_reason": "..."
}
```
- `score` + `summary` 仅 completed 时有值
- 前端根据已收到的事件计数计算整体进度（如 7/10 维度）

**`report_ready`** — 综合报告生成完成
```json
{
  "version": 1,
  "total_score": 72.5,
  "risk_level": "high"
}
```

##### 兜底设计原则

所有环节必须有失败时的处理策略:

| 场景 | 兜底方案 |
|------|---------|
| LLM 调用超时/失败 | 重试 2 次，仍失败则该维度仅展示程序化结果，标记"AI 研判暂不可用" |
| 报价解析规则批量应用失败 | 对失败的投标人单独调用 LLM 重新理解表结构；仍失败则标记为"报价解析失败"，该投标人价格相关维度跳过并在报告中说明 |
| 文档角色分类不确定 | LLM 标记为"待确认"，前端提示用户手动修正 |
| 某维度 Agent 整体崩溃 | 不影响其他 Agent，该维度标记为"检测失败"并记录原因 |
| 综合研判 LLM 失败 | 仅展示各维度的程序化评分和数据，不生成自然语言结论 |

#### 4.4.2 比对模式

| 模式 | 说明 | 适用维度 |
|------|------|---------|
| 两两比对 | 对项目内所有投标人两两组合 C(n,2)，逐对比较 | 硬件特征、价格构成、错误一致性、文本相似度、图片复用、软件特征、语言风格 |
| 整体分析 | 将所有投标人数据放在一起全局分析 | 报价规律性、接近最高限价、操作时间异常 |

#### 4.4.3 文档对齐原则

两两比对时，**仅对相同角色的文档进行比较**。即 A 的技术方案只与 B 的技术方案比，不与 B 的报价清单比。若某一方缺少对应角色的文档，该维度跳过。

```
投标人A               投标人B
技术方案.docx   ←→    B公司技术标.docx     (角色匹配: 技术方案)
报价清单.xlsx   ←→    报价表.xlsx          (角色匹配: 报价清单)
资质证明.docx   ←→    (无对应文件)          → 跳过
企业介绍.docx   ←→    公司简介.docx        (角色匹配: 企业介绍)
```

#### 4.4.4 检测维度

##### F-DA-01 硬件特征码雷同 [高风险] [两两比对] [铁证级]

- **提取策略（三层）**:
  1. 程序提取 DOCX custom properties（自定义属性字段中的机器标识）
  2. 程序扫描 XLSX 隐藏 sheet、特定命名 sheet（如"软件信息""系统信息"），以及关键词（"加密锁号""MAC地址""序列号"等）相邻单元格的值
  3. LLM 兜底：从文档正文中识别"投标文件制作信息"等段落，提取硬件标识（仅在前两步无结果时触发）
- **程序负责**: 按上述三层策略提取 MAC 地址、CPU 序列号、硬盘序列号、主板号、加密锁号，然后精确匹配
- **LLM 负责**: 第三层兜底提取
- **触发条件**: 同一标段，≥2家投标人任意一项硬件信息相同
- **铁证规则**: 命中即直接标记该投标人对为高风险，不依赖加权评分
- **输出证据**: 匹配的硬件字段名称、字段值、涉及的文件名、提取来源（属性/单元格/正文）

##### F-DA-02 错误一致性 [高风险] [两两比对] [铁证级]

- **程序负责**: 在 A 的全文中搜索 B 的标识信息（公司全称、简称、关键人员姓名、资质编号等，来源于解析阶段 LLM 提取结果），进行关键词交叉初筛。**防碰撞策略**：(1) 过短的标识词（≤2字符，如"中建""工程"）不单独作为搜索词，仅在与其他标识词共现时才纳入；(2) 单个标识词命中段落数超过阈值（默认50段）时自动降权，避免高频词引发 LLM context 溢出；(3) 发送给 LLM 的候选段落总数上限为 100 段，超出时按命中密度排序截断
- **LLM 负责**: 深度阅读程序标记的可疑段落，判断是否存在真正的交叉污染。LLM 能发现程序无法发现的隐蔽错误，如:
  - A 标书里出现 B 的项目经理姓名但未提公司名
  - A 和 B 引用了同一个特定项目案例但用了不同措辞
  - A 和 B 文件包含同一个罕见错别字或格式错误
  - 比对不同投标人文档中的图片 pHash，发现共用相同瑕疵图片
- **触发条件**: LLM 确认存在真正的交叉污染
- **铁证规则**: LLM 确认为真时，直接标记该投标人对为高风险
- **输出证据**: 混淆文本片段及位置、LLM 判断理由、共用图片列表

##### F-DA-03 文本相似度超标 [中风险] [两两比对]

- **程序负责**: 对同角色文档进行段落级 TF-IDF + 余弦相似度计算，标记相似度超阈值的段落对
- **LLM 负责**: 审阅程序标记的高相似段落，判断是模板雷同、行业通用表述，还是真正同源抄袭
- **比对粒度**: 段落级，段落划分规则:
  - 以 DOCX 原生段落（paragraph 对象）为基本单元
  - 过滤 < 50 字的短段落（标题、编号行等不参与相似度计算）
  - 表格内文本按行合并为一个比对单元
  - 50 字阈值管理员可配置
- **触发条件**: 两两比对中存在大量高相似段落（>85%），且 LLM 判断为非正常雷同
- **输出证据**: 高相似段落列表（含原文摘要、段落定位）、LLM 判断结论、整体相似度分数

##### F-DA-04 价格构成雷同 [中风险] [两两比对]

- **清单对齐策略**:
  1. 先按项目编码精确匹配
  2. 对未精确匹配到的剩余项目，用编码局部匹配（如前缀相同）+ 项目名称相似度联合补充对齐
  3. 仍无法对齐的项目，LLM 辅助判断对应关系
  4. 最终兜底：未能对齐的项目标记跳过；若对齐率过低（整体 < 50%），该 Pair 的价格构成维度整体跳过并在报告中说明原因
- **程序负责**: 对齐后进行两层检测:
  - 逐项比对: 计算单价/合价的余弦相似度，找出一致项
  - 分布模式比对: 检测整体报价分布趋势是否一致（防止个别项目微调但整体趋势相同的规避手段）
- **LLM 负责**: 判断高相似是正常（定额标准/政府指导价导致）还是异常串通
- **触发条件**: 逐项相似度 > 95% 或分布模式高度一致，且 LLM 判断为非正常一致
- **输出证据**: 清单对齐方式、逐项相似度分数、分布模式分析结果、相同报价项列表、LLM 判断理由

##### F-DA-05 图片/图表复用 [中风险] [两两比对]

- **程序负责**: 比对不同投标人文档中提取的图片哈希值（MD5 精确匹配 + pHash 模糊匹配，汉明距离 < 5）
- **LLM 负责**: 判断重复图片是通用素材（logo、模板装饰图、招标方提供的图纸）还是应该各自独立制作的内容（施工方案图、组织架构图等）
- **触发条件**: 不同投标人标书中存在哈希相同的非通用图片
- **升级规则**: 如果 LLM 判断为"应各自独立制作"的图片 MD5 相同，可升级为铁证级
- **输出证据**: 复用图片列表、图片所在文件及位置、LLM 判断（通用/非通用）

##### F-DA-06 语言风格一致性 [中风险] [两两比对] [LLM 独有维度]

- **LLM 负责**: 抽取各投标人技术方案等文本的代表性段落，分析写作风格是否同源——用词偏好、句式结构、标点习惯、段落组织方式等
- **程序不参与**: 此维度完全依赖 LLM 的语言理解能力
- **调用策略**: 先对每个投标人提取写作风格特征摘要（N 次调用），再全局一次比较（1 次调用）。当投标人数量 > 20 时，自动退化为分组比较以避免超出 context 窗口
- **触发条件**: LLM 判断两个投标人的写作风格高度一致
- **局限性说明**: 风格一致可能源于同一主体操控，也可能源于委托了同一代写服务。Agent 输出结果时必须主动说明此局限性，建议用户结合其他维度综合判断
- **输出证据**: LLM 的风格分析报告、典型相似风格片段对比、局限性说明

##### F-DA-07 软件特征一致 [中风险] [两两比对]

- **程序负责**: 比对文档元数据中的作者、保存者、公司名称、创建软件、计价锁号等字段（精确匹配 + 白名单过滤）
- **白名单过滤**: Administrator、User 等通用默认值不算有效匹配
- **LLM 不参与**
- **触发条件**: 同一标段，≥2家投标人元数据高度一致或计价锁号相同
- **输出证据**: 匹配的元数据字段明细

##### F-DA-08 操作时间异常 [中风险] [整体分析]

- **程序负责**: 收集所有投标人所有文档的创建时间和修改时间，进行两项检测:
  - 时间窗口聚集检测: 将所有时间戳排序，用滑动窗口（默认 30 分钟）扫描，检测窗口内是否有 ≥ 3 个不同投标人的时间戳
  - 非工作时间集中度: 统计 0:00-6:00 时段内的时间戳占比
  - 30 分钟窗口大小和 3 个投标人阈值均管理员可配置
- **LLM 不参与**
- **触发条件**: 滑动窗口内出现 ≥ 3 个不同投标人的时间戳，或非工作时间时间戳占比异常偏高
- **可靠性说明**: 文档时间戳可被篡改，此维度可靠性有限，报告中需说明此局限性
- **输出证据**: 时间聚类结果、异常时间窗口（起止时间+涉及投标人）、非工作时间占比

##### F-DA-09 报价规律性差异 [中风险] [整体分析]

- **程序负责**: 将所有投标人的总报价排序，检测等差数列、等比数列、固定百分比递减/递增等数学规律
- **LLM 不参与**
- **触发条件**: 报价序列呈显著线性相关 (R² > 0.95) 或几何相关性
- **输出证据**: 规律类型（等差/等比/百分比）、拟合度、报价分布图

##### F-DA-10 异常接近最高限价 [中风险] [整体分析]

- **程序负责**: 计算所有有效投标报价相对于最高限价的偏离度和方差
- **LLM 不参与**
- **前置条件**: 项目已填写最高限价/控制价
- **触发条件**: 所有有效投标报价的方差趋近于零，且均在最高限价的 98%-100% 区间内
- **输出证据**: 方差值、各投标人报价与限价的偏离百分比、报价分布图

### 4.5 评分与报告

#### F-RP-01 综合评分模型

评分采用**两层结构**:

##### 第一层: 投标人对（Pair）级别评分

每一对投标人 (A×B) 有独立的风险评分，这是最小判断单元。

| 维度 | 权重 | 性质 |
|------|------|------|
| 硬件特征码雷同 | 20 | 铁证级 |
| 错误一致性 | 20 | 铁证级 |
| 文本相似度超标 | 15 | 加权计算 |
| 价格构成雷同 | 15 | 加权计算 |
| 图片/图表复用 | 13 | 加权计算（LLM 判断非通用图时可升为铁证级） |
| 语言风格一致性 | 10 | 加权计算 |
| 软件特征一致 | 7 | 加权计算 |
| **合计** | **100** | |

**Pair 评分公式**:
```
pair_score = Σ(dimension_score_i × weight_i) / 100
```
- 每个 Agent 输出 0-100 的维度得分（0=无异常，100=最大异常）
- 权重之和 = 100，公式结果范围 0-100
- Agent 跳过或失败的维度，该维度得分计为 0（不拖高风险判定）

**铁证直判规则**: 如果任意铁证级维度命中（硬件特征码相同、LLM 确认错误一致性为真、非通用图片 MD5 相同），该投标人对直接标记为高风险（is_iron_evidence=true），不依赖加权评分总分。铁证 Pair 的 pair_score 仍正常计算（用于排序和展示），但风险判定不依赖此分数。

##### 第二层: 项目级别评分

```
project_score = min(max(所有 pair_score) + overall_bonus, 100)

overall_bonus = Σ(触发的整体维度 × 5)，上限 15 分:
  · 报价规律性异常触发: +5
  · 异常接近限价触发: +5
  · 操作时间异常触发: +5
```
- 封顶 100 分（max pair_score + overall_bonus 超过 100 时截断）
- 整体维度未触发时加分为 0，跳过/失败的整体维度不加分

##### 风险判定

- **高风险 (project_score ≥ 70 或 任意 Pair 铁证命中)**: 强烈疑似围标，建议重点审查
- **中风险 (40 ≤ project_score < 70)**: 存在围标嫌疑，建议人工复核
- **低风险 (<40分)**: 未发现明显围标特征

#### F-RP-02 检测报告

报告内容包括:

1. **项目概览**: 项目信息、投标人数量、检测时间
2. **风险概要**: 综合评分、风险等级、LLM 生成的综合风险结论（自然语言，站在项目整体视角，不逐对罗列）
3. **维度雷达图**: 各维度得分分布可视化
4. **高风险投标人对**: 列出最可疑的投标人组合及关联证据
5. **各维度明细**:
   - 是否触发预警，或标记为"已跳过"并说明原因
   - 具体检测数据（分数、匹配项列表）
   - LLM 判断结论和理由（如适用）
   - 维度局限性说明（如适用：操作时间可篡改、语言风格可能源于代写等）
   - 证据文本摘要

**第一期: 结论报告**（给决策者看）

## 5. 非功能需求

### 5.1 性能

| 指标 | 要求 |
|------|------|
| 单个投标人文件包解析 | ≤ 5 分钟 (100MB以内压缩包) |
| 10个投标人完整检测分析 | ≤ 15 分钟 |
| 页面响应时间 | ≤ 2 秒 (常规操作) |
| 并发支持 | 至少 5 个项目同时分析 |

### 5.2 文件处理容量

| 指标 | 要求 |
|------|------|
| 单项目最大投标人数 | 50 |
| 单个压缩包大小上限 | 500MB |
| 单个投标人解压后文件数上限 | 1000 |
| 支持的最大单文件大小 | 100MB |

### 5.3 可靠性

- 解析失败不影响其他文件和投标人的处理
- 单个维度检测失败不影响其他维度的结果
- LLM 调用失败有重试机制（重试 2 次），不阻塞程序化分析结果
- 异步任务具备超时保护（单 Agent 5 分钟，全局 30 分钟）。**进程级超时终止**：asyncio 的 `task.cancel()` 无法终止已提交到 ProcessPoolExecutor 的子进程，超时后必须通过 `Process.kill()` 强制终止卡死的 worker 进程并回收资源，防止僵尸进程累积导致 OOM
- **已知限制**: 第一期使用 asyncio 进程内任务，进程重启后未完成的解析/检测任务丢失，需用户手动重新触发
- **已知限制**: 投标人数量较多时（如 >20），综合结论生成的 LLM 输入（所有 Pair 结果汇总）可能接近 context 窗口上限。第一期投标人数量有限，暂不处理；如将来出现需对输入做摘要压缩

### 5.4 安全性

- 上传文件存储在服务端隔离目录，不同项目之间隔离
- 文件下载需鉴权
- 防 zip bomb（解压大小和文件数量限制）
- 防 Zip Slip 目录穿越（解压路径校验，拒绝 `..` 和绝对路径）
- 上传文件类型校验（不仅检查扩展名，还检查文件头魔数）
- LLM API Key 服务端管理（第一期通过环境变量，第二期加密存储在数据库），前端不接触
- 投标文件内容发送至 LLM API 的范围明确可控（见 12.3 LLM 数据安全说明）

### 5.5 数据生命周期

数据分为两类，生命周期不同:

**到期清理（原始文件）:**
- 上传的压缩包
- 解压后的 DOCX/XLSX 原始文件
- 提取的图片文件本身（磁盘文件）
- 项目完成复核后保留 N 天（默认 90 天，管理员可配置），到期自动清理

**长期保留（解析产物 + 分析结果）:**
- DocumentText（全文文本、段落数组）
- DocumentMetadata（元数据字段）
- PriceItem（报价明细）
- DocumentImage（图片哈希值和位置，不含图片文件本身）
- PairComparison / OverallAnalysis（比对结果）
- AnalysisReport（检测报告）
- AgentTask（Agent 运行记录）

保留期内审查员可下载原文件和使用对比视图；过期后原文件不可下载，但对比视图仍可使用（基于长期保留的解析产物）。

注: 解析产物（文本、报价数据）会持续占用数据库存储，管理员应定期关注数据库容量。清理操作记录日志，可审计。

## 6. 文件处理边界情况

| 场景 | 处理策略 |
|------|---------|
| 压缩包嵌套压缩包 | 递归解压，最大深度 3 层 |
| 加密压缩包 | 标记为"需密码"，支持用户输入密码后重试 |
| DOC/XLS/WPS 旧格式 | 标记为"格式不支持"，提示用户转换为 DOCX/XLSX 后重新上传 |
| 超大文件 (>100MB) | 标记警告，尝试分块处理，设超时保护 |
| 文件名编码乱码 | 依次尝试 UTF-8 / GBK / GB2312 解码 |
| 空文件或损坏文件 | 标记为"解析失败"并记录原因，不影响其他文件 |
| 同一投标人多次上传 | 追加合并到同一投标人下 |
| 重复文件 | 基于文件 MD5 去重，提示用户 |

## 7. 交叉比对模型

### 7.1 比对组合

假设项目有 n 个投标人:

- **两两比对维度** (7个): 产生 C(n,2) = n×(n-1)/2 组比对
- **整体分析维度** (3个): 对所有 n 个投标人整体分析

示例: 10个投标人 → 45对两两比对 × 7个维度 = 315 个两两比对任务 + 3 个整体分析任务

### 7.2 文档对齐

两两比对时遵循"同角色对齐"原则（见 4.4.3）。

### 7.3 任务调度

各 Agent 不是按阶段整体启动，而是各自前置条件满足即启动（详见 4.4.1 Agent 依赖关系图）。

整体流程:
1. 至少 2 个投标人的解析完成后，满足条件的 Agent 立即启动
2. 各 Agent 并行运行，完成一个推送一个结果
3. 全部 Agent 完成或超时后，LLM 综合研判生成结论报告

渐进式出结果: 每个 Agent 完成后通过 SSE 实时推送结果到前端。

### 7.4 增量检测 [第二期]

> **第一期**: 追加投标人后点击"开始检测"，系统执行全量重新分析（所有投标人配对全部重跑），生成新版本报告。简单可靠，无需区分新旧配对。

**第二期增量检测方案**（供参考，第一期不实现）:

- **两两比对维度**: 仅运行新增投标人与已有投标人的交叉比对，已完成的 Pair 不重复分析
- **整体分析维度**: 需全量重跑（因为投标人集合变了，报价规律等需要重新计算）
- **评分**: 基于全量 PairComparison + 新的 OverallAnalysis 全量重算
- **报告**: 生成新版本报告（version 递增），旧版本保留不覆盖
- **Agent 状态**: 未重跑的维度从上一版本复制 AgentTask 记录，标记为"沿用上次结果"

- `doc_role_keywords`: 文档角色关键词（LLM 不可用时的兜底匹配）
- `hardware_keywords`: 硬件信息扫描关键词（XLSX 单元格扫描用）
- `metadata_whitelist`: 元数据通用值白名单（匹配时过滤）
- `min_paragraph_length`: 短段落过滤阈值（低于此字数不参与相似度计算）
- `file_retention_days`: 原始文件保留天数（到期自动清理）
- 超时/重试等运维参数通过环境变量配置，不放在此 JSON 中

## 9. 前端设计

### 9.1 设计规范

| 项 | 规范 |
|---|------|
| 目标分辨率 | 1920×1080 最小宽度，适配更大屏幕 |
| 布局 | 左侧固定导航（可收起）+ 内容区自适应宽度 |
| 信息密度 | 偏高密度，表格优先于卡片，减少无意义留白 |
| 风险色彩体系 | 🔴 红(高风险/铁证) / 🟡 橙(中风险/触发) / 🟢 绿(低风险) / ⚪ 灰(未触发/跳过) |
| 铁证标记 | 红色背景 + 加粗文字 + "铁证"标签，在任何出现的地方都醒目 |
| 字体 | 正文 14px，表格 13px，标题按层级递减 |
| 组件库 | Ant Design，不做大量自定义，保持一致性 |
| 基础模板 | Ant Design Pro |

### 9.2 设计原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | 为大屏设计 | 充分利用横向空间，善用分栏、表格、侧边面板。不做移动端纵向堆叠布局 |
| 2 | 操作轻、消费重 | 上传/配置页极简聚焦；报告页信息丰富但分层展示 |
| 3 | 操作和消费分离 | 项目详情页专注操作（上传、管理投标人、启动检测），报告页专注消费（看结论、看证据、做对比），不混在一个页面 |
| 4 | 检测进度要轻 | 进度条 + 一行最新动态摘要即可，不做重型实时面板。Agent 详细运行记录归入报告的"检测日志"Tab |
| 5 | 报告分 Tab 不堆叠 | 用 Tab 分区（概要/维度明细/投标人对详情/检测日志），不做超长滚动页 |
| 6 | 表格优先于卡片 | 投标人列表、文件列表、报价明细、检测日志等均用表格。卡片仅用于项目列表（项目少，需要视觉区分风险等级） |
| 7 | 关系矩阵是核心交互入口 | 报告页的投标人×投标人热力图是核心导航组件，点击单元格钻取该 Pair 的各维度详情 |
| 8 | 渐进式反馈 | 解析和检测过程中，完成一步展示一步，通过 SSE 实时推送（见 4.4.1）。用户应能在任何阶段感知系统在做什么、做到了哪里、大致结果如何 |
| 9 | 操作极简化 | 用户不需要理解检测维度的技术细节。启动检测只需一个按钮。复杂决策交给系统（Agent 自判断），简单确认交给用户（如文档角色修正） |
| 10 | 表单防丢失 | JWT 过期（401）时，前端 axios 拦截器若检测到当前页面有未提交表单数据（如复核意见），先缓存到 localStorage，登录成功后自动恢复。避免长时间填写内容因 token 过期而丢失 |

## 10. API 接口设计

> 详细的请求/响应字段由 FastAPI Pydantic schema 定义，运行时自动生成 OpenAPI 文档。本节定义端点路径、方法、关键参数和错误码。
> 权限标注：🔓=需登录，👤=仅本人项目（管理员可只读访问他人项目：查看详情/报告/导出，但不可执行写操作），👑=仅管理员

### 10.1 认证

| 方法 | 路径 | 说明 | 关键参数/响应 | 错误码 |
|------|------|------|-------------|--------|
| POST | `/api/auth/login` | 登录 | 请求: {username, password} → 响应: {access_token, token_type, user} | 401 凭证错误, 403 已禁用, 429 已锁定 |
| GET | `/api/auth/me` | 获取当前用户 🔓 | 响应: {id, username, role, must_change_password} | 401 未登录/过期 |
| POST | `/api/auth/change-password` | 修改密码 🔓 | 请求: {old_password, new_password} | 400 旧密码错误, 422 新密码不满足规则 |

### 10.2 项目管理

| 方法 | 路径 | 说明 | 关键参数/响应 | 错误码 |
|------|------|------|-------------|--------|
| GET | `/api/projects` | 项目列表 🔓 | 查询: ?status=&risk_level=&search=&page=&size= → 响应: {items[], total, page, size} | - |
| POST | `/api/projects` | 创建项目 🔓 | 请求: {name, bid_code?, max_price?, description?} → 响应: Project 对象 | 422 验证失败 |
| GET | `/api/projects/{id}` | 项目详情 🔓👤 | 响应: Project + bidders[] + reports[] + current_analysis?。当 status=analyzing 时附带 current_analysis: {version, started_at, agent_tasks_summary: {total, completed, running, skipped, failed}}，用于前端展示进行中检测的进度行 | 404 不存在/无权限 |
| DELETE | `/api/projects/{id}` | 删除项目 🔓👤 | 级联删除所有关联数据 + 磁盘文件 | 404, 409 检测进行中 |

### 10.3 投标人与文件

| 方法 | 路径 | 说明 | 关键参数/响应 | 错误码 |
|------|------|------|-------------|--------|
| POST | `/api/projects/{pid}/bidders` | 添加投标人 🔓👤 | 请求: multipart {name, file?} → 响应: Bidder 对象 | 409 名称重复, 413 文件过大, 415 格式不支持 |

| DELETE | `/api/projects/{pid}/bidders/{bid}` | 删除投标人 🔓👤 | 级联删除文件和分析结果 | 409 检测进行中 |
| POST | `/api/projects/{pid}/bidders/{bid}/upload` | 追加上传 🔓👤 | 请求: multipart {file} → 响应: {new_files, skipped_duplicates} | 413, 415 |
| GET | `/api/projects/{pid}/bidders/{bid}/documents` | 文件列表 🔓👤 | 响应: BidDocument[] 含 doc_role, parse_status, parse_error | - |

## 10.8 LLM 调用规格定义

> 所有语义判断类调用统一输出结构化 JSON，包含 `score`（0-100）、`reasoning`（推理过程）、`summary`（一句话结论）。以下定义各场景的输入内容、调用策略和兜底方案。

### 解析阶段

#### L-1 文档角色分类 + 标识提取

- **调用策略**: 每个投标人 1 次调用（N 个投标人 = N 次）
- **输入**:
  - 投标人名称
  - 文件列表：每个文件的文件名、文件类型（docx/xlsx/image）、首段文本（前 200 字）
- **输出**:
  - `roles[]`：每个文件的角色分类（9 种之一）+ 置信度（high/low）
  - `identity_info`：公司全称、简称、关键人员姓名列表、资质编号列表、联系方式
- **兜底**: LLM 失败时用文件名关键词匹配角色（关键词列表管理员可配置），标识信息留空（错误一致性维度降级为纯关键词匹配）

#### L-2 报价表结构识别

- **调用策略**: 每个项目 1 次调用（招标方统一模板，识别一次后批量应用）
- **输入**:
  - 第一个报价类文件的 sheet 名列表
  - 每个 sheet 的前 10 行原始单元格数据
- **输出**:
  - 目标 sheet 名称
  - 表头行号、数据起始行号
  - 列映射（哪列是项目编码/项目名称/单位/数量/单价/合价）
- **兜底**: 识别失败标记"无法识别报价表结构"，前端提示用户手动配置列映射

#### L-3 硬件信息提取（第三层兜底）

- **调用策略**: 仅在程序提取（属性字段 + 单元格扫描）无结果时触发，按投标人调用
- **输入**:
  - 该投标人所有文档中包含硬件相关关键词（锁号、MAC、序列号等）的上下文段落（关键词前后各 200 字）
- **输出**:
  - `hardware_items[]`：每项含类型（MAC/CPU/硬盘/主板/锁号）、值、来源段落摘要
- **兜底**: LLM 也提取不到则该投标人无硬件信息，硬件特征维度对其不适用

### 检测阶段

#### L-4 文本相似度判断

- **调用策略**: 按 Pair × 文档角色调用（程序先筛出超阈值段落对，仅发高相似的）
- **输入**:
  - 投标人 A 和 B 的名称
  - 高相似段落对列表：每对含 A 原文、B 原文、程序计算的相似度分数
  - 文档角色（如"技术方案"）
- **输出**:
  - 每对段落的判断：`template`（模板雷同）/ `generic`（行业通用表述）/ `plagiarism`（同源抄袭）
  - 整体结论 + 置信度
- **兜底**: LLM 失败时只展示程序相似度分数，标记"AI 研判暂不可用"

#### L-5 错误一致性判断

- **调用策略**: 按 Pair 调用，1 次调用包含双向检查结果
- **输入**:
  - 投标人 A 和 B 的标识信息清单
  - A 文档中含 B 标识信息的可疑段落（程序关键词初筛结果，已执行防碰撞过滤：短词不单独匹配、高频词降权、总段落数≤100）
  - B 文档中含 A 标识信息的可疑段落（同上，双向）
  - A 和 B 的页眉页脚文本（用于检测公司名混淆）
- **输出**:
  - 是否存在真正的交叉污染（bool）
  - `evidence[]`：每条含污染类型（公司名混入/人员名混入/案例共用/错别字共用）、原文片段、位置
  - 铁证标记（`direct_evidence`: true/false）
- **兜底（LLM 失败）**: 仅展示关键词命中结果，不做铁证判定，标记"AI 研判暂不可用"
- **兜底（identity_info 为空）**: 不跳过，降级为用投标人名称（Bidder.name）做关键词交叉搜索。降级模式不做铁证判定，标记"降级检测，标识信息不足"

#### L-6 价格合理性判断

- **调用策略**: 按 Pair 调用
- **输入**:
  - 逐项相似度分数 + 完全一致的报价项清单（项目名称 + 单价）
  - 分布模式分析结果（R² 值、趋势描述）
  - 项目基本信息（行业、项目类型——辅助判断是否有政府指导价）
- **输出**:
  - 判断：`normal`（定额/指导价导致的正常一致）/ `suspicious`（异常串通）
  - 判断理由
- **兜底**: LLM 失败时只展示程序数值分析结果

#### L-7 图片通用性判断

- **调用策略**: 按重复图片组调用（相同哈希的图片归为一组，每组 1 次调用），结果复用到所有涉及该图片的 Pair
- **输入**:
  - 重复图片涉及的投标人列表
  - 每个投标人中该图片所在的文档角色、文档位置描述
  - 图片前后的上下文文本（各 100 字）
- **输出**:
  - 判断：`generic`（通用素材：logo/模板图/招标方图纸）/ `specific`（应独立制作：方案图/架构图等）
  - 判断理由
  - 若判断为 `specific` 且 MD5 相同，标记可升级为铁证级
- **兜底**: LLM 失败时所有重复图片标记为"待人工判断"，不自动升级铁证

#### L-8 语言风格分析

- **调用策略**: 两阶段调用
  - 第一阶段：每个投标人 1 次（提取风格特征）
  - 第二阶段：1 次全局比较（>20 投标人时自动分组）
- **第一阶段输入**:
  - 投标人名称
  - 技术方案类文档的代表性段落（程序抽样 5-10 段，每段 100-300 字）。**抽样策略**：先用 TF-IDF 过滤掉高频通用段落（如法规条文、行业标准模板），再从剩余段落中均匀抽样，确保捞取的是投标人原创内容而非行业共性废话
- **第一阶段输出**:
  - 风格特征摘要：用词偏好、句式特点、标点习惯、段落组织方式
- **第二阶段输入**:
  - 所有投标人的风格特征摘要（不含原文，控制 context 大小）
- **第二阶段输出**:
  - 风格高度一致的投标人组合列表 + 典型相似特征描述
  - 局限性说明（风格一致可能源于同一代写服务）
- **兜底**: 任一阶段 LLM 失败则整个维度跳过，标记"语言风格分析不可用"

### 12.2 系统架构图

```
[浏览器] → [Nginx] → [FastAPI 后端] → [PostgreSQL]
                           │
                     asyncio 协程 + ProcessPoolExecutor
                           │
                     ┌─────┴─────┐
                     ↓           ↓
               [本地文件存储]  [LLM API]
```

### 12.3 LLM 调用架构

```
程序化分析 (确定性计算)          LLM (理解与判断)
─────────────────────         ────────────────────
元数据提取 + 精确匹配           文档角色批量分类
图片哈希计算 + 比对             报价表结构理解 (每项目1次)
TF-IDF 段落相似度计算           高相似段落同源性判断
报价数值相似度计算              价格异常合理性判断
时间戳聚类分析                 错误一致性深度判断
数列规律检验                   语言风格一致性分析
方差统计                      综合结论生成
```

#### LLM 调用层设计

- **模型无关架构**: 统一 LLM 调用接口，通过配置切换供应商（Claude / GPT / 通义千问 / DeepSeek 等）
- **密钥管理**: 第一期通过环境变量（.env 文件）管理 API Key、Endpoint，服务端加载，前端不接触。第二期迁移到数据库加密存储 + 管理界面
- **Prompt 管理**: 第一期 Prompt 模板在代码中维护（Python 模块或 YAML 文件），修改需发布部署。第二期迁移到数据库，支持管理后台在线编辑、版本历史和一键回滚
- **输出格式**: 语义判断类 Agent 统一输出结构化 JSON，包含 direct_evidence（铁证标记）、score（置信度分数）、reasoning（推理过程）、summary（一句话结论）

#### LLM 数据安全说明

以下场景会将投标文件内容发送至 LLM API:

| Agent | 发送内容 | 敏感度 |
|-------|---------|--------|
| 文档角色分类 | 文件名 + 首段文本 | 低 |
| 报价表结构识别 | Excel 前几行数据 | 中 |
| 文本相似度判断 | 高相似段落原文 | 高 |
| 错误一致性判断 | 可疑段落原文 | 高 |
| 价格合理性判断 | 相似度分数 + 异常项列表 | 中 |
| 图片通用性判断 | 图片上下文描述 | 低 |
| 语言风格分析 | 代表性段落原文 | 高 |
| 硬件信息提取 | 文本片段 | 低 |
| 综合结论生成 | 程序预聚合的分析摘要（铁证发现、高风险群组、各维度概览），不含原始文档内容 | 低 |

管理后台"规则配置"中提供选项，允许关闭特定 Agent 的 LLM 调用（退化为纯程序分析），供数据安全要求较高的场景使用。
