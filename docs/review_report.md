# 围标检测系统需求及设计审查报告 (V2 Forensic Audit)

**审查对象:** 需求文档 (`requirements.md`) & 用户故事 (`user-stories.md`)
**审查人:** Antigravity (Advanced Agentic AI)
**状态等级:** 🔴 拒绝执行 (状态: 需重构核心架构)
**前言:** 如果仅仅是修修补补，那是普通代码助手的标准。作为顶级架构审查，第一轮指出的 API 错位和并发只是表象。如果你需要触及灵魂的拷问，以下是我在这 3000 多行文档的深水区挖掘出的**安全级别的漏洞、进程级别的内存泄漏灾难、以及伪逻辑导致的误报风暴**。

Opus 的这份设计在理想的顺境下是一份好蓝图，但在真实的边缘场景（Edge Cases）和对抗性输入下，这套系统**活不过上线后的第一个高压测试周**。

以下是真正的降维打击问题清单：

---

## 1. 致命的系统安全与稳定性灾难 (FATAL SYSTEM CRASHERS)

### 1.1 经典的 Zip Slip 目录穿越漏洞 (Security CVE 级漏洞)
* **发现位置:** F-FM-02 / US-4.1
* **深层问题:** 防御 Zip Bomb 时，文档仅规定了"限制总大小 2GB 和 1000 个文件"。但它**完全没有提及 Zip 内路径清洗（Path Sanitization）**！攻击者可以轻易构造内含 `../../../../etc/shadow` 或 `../../../../root/.ssh/authorized_keys` 路径的恶意压缩包。当你的后台 Python 脚本甚至以不当权限解压时，将会直接导致**服务器任意文件覆写**。这是最低级、但在 AI 自动生成的需求中最容易漏掉的系统级安全漏洞。

### 1.2 Asyncio 取消机制引发的"僵尸进程"与 OOM (Memory Leak)
* **发现位置:** TS-1 / `requirements.md` 12.3
* **深层问题:** 设计声称"Agent 具有单任务 5 分钟超时保护...CPU 密集任务使用 ProcessPoolExecutor"。
* **技术盲区:** Python `asyncio` 的 `task.cancel()` **根本无法杀死**正在 `ProcessPoolExecutor` 中运行的子进程！如果提取哈希值或 TF-IDF 的任务发生死循环或超时，虽然主 API 协程返回了 "timeout"，但后台那个满载 CPU 的 Python 进程将永远卡死在后台。10 次超时就会产生 10 个满载 100% 的幽灵进程，最终导致整个服务器 OOM（内存溢出）和死机。

### 1.3 `PriceParsingRule` 并发锁的"白日梦"设计 (Race Condition)
* **发现位置:** F-PA-05 步骤4 / US-4.4 AC5
* **深层问题:** "多个投标人同时到达步骤 4 时，仅第一个执行，后续等待规则生成后直接应用。"
* **技术盲区:** 在没有任何分布式锁（如 Redis / ZooKeeper）且各任务是独立运行的 `ProcessPoolExecutor` 或协程的架构下，多个异构任务如何实现"等待"另一个任务的结果？靠轮询数据库（Polling）会造成微服务雪崩，靠内存锁跨进程会直接失效。整个 F-PA-05 的并发控制缺乏实现路径，是经典的"PPT级架构"。

## 2. API 闭环的离谱缺失 (MIND-BOGGLING OMISSIONS)

### 2.1 "看得见摸不着"的幽灵文件 (Missing Core API)
* **发现位置:** `requirements.md` 10 节（API 接口设计）
* **深层问题:** 整个系统要求审查员上传海量 PDF/DOCX 进行解析，前端也设计了“文件列表”页面。然而，遍览文档第 10 节长达几十个 API——**竟！然！没！有！设！计！文！件！下！载！接！口！** (无 `GET /api/documents/{id}/download`)。
审查员在发现异常后，甚至无法把那份被判定为抄袭的原始投标文件下载下来进行证据保全。所谓的"全流程业务闭环"在这里直接断崖。

### 2.2 恢复重连后的"薛定谔进度条" (SSE Sync Failure)
* **发现位置:** F-PM-03 / TS-2
* **深层问题:** "SSE 断线重连后通过调 API 拉取最新状态"。拉取哪个 API？`GET /projects/{id}`。但文档中明确表明，该接口只返回 `{total, completed, running, skipped, failed}` 这种汇总计数！
一旦用户网络闪断了 5 秒钟重连，这 5 秒内 LLM 输出的**所有具体分数、结论摘要 (agent_status.summary)** 将永远消失在黑洞里，直到最终报告生成前，前端都无法恢复详细的进度流状态！

### 2.3 鉴权逻辑：强制登出的复核大清空 (UX Disaster)
* **发现位置:** US-1.1
* **深层问题:** 强行写死 24 小时过期且没有 Refresh Token (刷新令牌) 的双 Token 续签设计。当审查员花费 2 小时阅读了一份 500 页的标书、并在 P-08 页面的表单里写下了 3000 字的详细复核意见时——此时 token 过期了。审查员点击"提交"，系统因为 401 Unauthorized 直接将其踢回 `/login` 登录页，辛苦写就的 3000 字复核报告灰飞烟灭。这个用户体验堪称毁灭级。

## 3. LLM 伪逻辑与误报风暴 (LLM FALSE-POSITIVE STORMS)

### 3.1 单点故障产生蝴蝶效应：铁证据"隐性失效"
* **发现位置:** 文档分类兜底 L-1 / 错误一致性 L-5
* **深层问题:** 如果 L-1 阶段因为 LLM 超时而走到兜底逻辑（标识信息留空）。那么后续最核心的"铁证维度"错误一致性 Agent（L-5）的前置条件就会缺失。
* **致命后果:** 这是整个防围标系统的招牌能力。然而，仅仅因为解析第一步 API 请求超时没有拿到公司名称，整个雷达系统竟然会对其"永久静默"，不仅跳过检测，而且不触发任何警告级别的主动重连。一个 200 请求里可能发生 1 次的网络抖动，就能让真正的围标者畅通无阻。

### 3.2 "写作风格分析"的统计学骗局 (L-8 Language Style)
* **发现位置:** F-DA-06 
* **深层问题:** "从全文均匀抽样 5-10 段"。标书大部分内容是毫无特色的枯燥模板（如《国家建筑工程安全规定》、《质量保证体系》），且各个投标公司均直接 copy 行业标准。
均匀抽样会导致抓出的文本里塞满了**强同质化的样板废话**，导致 LLM 判断这 10 个公司的行文习惯"高度一致"，从而引发海量误报。风格抽样前如果没有基于 TF-IDF 将极高词频的通用句式剔除，这个维度就没有任何实战价值。

### 3.3 价格嵌套表抽取的降维（L-2 Price Extraction）
* **发现位置:** L-2 报价结构提示词 / `requirements.md` 8
* **深层问题:** 数据库的 `PriceItem` 定义了 `unit_price_detail`（嵌套的综合单价明细），这是复杂评标中最核心的造价串标藏身地！
但 L-2 的 LLM 设计仅要求大模型找到“表头行”和“数据行”进行“列映射”。面对真实的招投标数据里的**父子行、多级标题、跨表联查**情况，单向列映射架构就像是用牙签去挖隧道，注定无法落库。

### 3.4 Token 爆炸的定时炸弹 (Error Consistency LLM)
* **发现位置:** F-DA-02
* **深层问题:** "程序找 A 的标识在 B 中的可疑段落...发给 LLM"。
设想 A 公司的简称是 "中建" 或 "工程"。当 B 的标书由于行业属性大量使用 "工程" 一词时，程序通过简单的 Regex / 关键词初筛，会将数千个"工程"所在的段落打包全部发送给 LLM。这种极其容易碰撞的短标量，会瞬间让 LLM 发生 Context Overload (上下文溢出)，单次调用的费用都会达到惊人水准。缺乏**实体消歧（Entity Disambiguation）**与短文本排除算法的初筛，无异于引颈就戮。
