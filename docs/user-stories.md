# 围标检测系统 — User Stories（OpenSpec TDD 格式）

> **版本**: v0.1 Draft
> **日期**: 2026-04-13
> **说明**: 每个 Story 包含角色/功能/价值、验收标准(AC)、测试验证点(前端E2E + 后端API)。

## Epic 1: 认证与授权

### US-1.1 用户登录

**作为** 系统用户（审查员/管理员），
**我希望** 通过用户名和密码登录系统，
**以便** 获得授权访问系统功能。

**描述**:
登录页为系统唯一入口。用户输入用户名和密码，后端验证通过后返回 JWT token。前端存储 token 并跳转至项目列表页（审查员）或管理首页（管理员）。

**`[GAP]` 前后端对齐**:
- 后端: 当前无 auth 相关端点，需新增 `POST /api/auth/login`、`GET /api/auth/me`、`POST /api/auth/change-password`
- 后端: 需新增 User 模型（username, password_hash, role, is_active, login_fail_count, locked_until）
- 后端: 依赖已安装但未使用: python-jose (JWT), passlib (bcrypt)
- 前端: 无登录页面、无 token 存储逻辑、无路由库

**验收标准 (AC)**:
1. 用户输入正确的用户名和密码后，系统返回 JWT token 并跳转到首页
2. 用户名或密码错误时，显示"用户名或密码错误"提示，不泄露具体是哪个字段错误
3. 密码在数据库中以 bcrypt 哈希存储，永不明文
4. JWT token 包含 user_id、role、过期时间（默认24小时）
5. 连续5次登录失败后，账户锁定15分钟

**测试验证点**:
- **后端 API 测试**:
  - `POST /api/auth/login` 正确凭证 → 200 + `{access_token, token_type, user}`
  - `POST /api/auth/login` 错误密码 → 401 + 通用错误信息
  - `POST /api/auth/login` 不存在的用户 → 401（同样的通用错误信息）
  - `POST /api/auth/login` 已禁用用户 → 403
  - 连续5次错误 → 第6次返回 429
- **前端 E2E 测试**:
  - 访问任意页面未登录 → 重定向至 /login
  - 输入正确凭证 → 跳转至 /projects
  - 输入错误凭证 → 页面显示错误提示，停留在登录页
  - 刷新页面后 token 仍有效（localStorage 持久化）

### US-1.2 用户登出

**作为** 已登录用户，
**我希望** 点击登出按钮退出系统，
**以便** 安全结束当前会话。

**描述**:
用户点击右上角的登出按钮，前端清除 token 和用户状态，跳转回登录页。

**验收标准 (AC)**:
1. 点击登出后，前端清除 localStorage 中的 token
2. 清除后访问任何受保护页面均重定向至登录页
3. 登出后使用旧 token 调用 API 返回 401

**测试验证点**:
- **前端 E2E 测试**:
  - 点击登出 → localStorage 无 token → 自动跳转 /login
  - 登出后浏览器后退按钮 → 不能回到受保护页面

### US-1.3 路由守卫与权限控制

**作为** 系统，
**我希望** 根据用户角色控制页面访问权限，
**以便** 确保审查员不能访问管理功能，未登录用户不能访问系统。

**`[GAP]` 前后端对齐**:
- 后端: 需实现 JWT 中间件依赖注入（`get_current_user`），所有路由需添加权限装饰器
- 前端: 需安装 react-router-dom，实现 `<ProtectedRoute>` 组件

**描述**:
系统分为三层访问控制：
- 未登录 → 只能访问 /login
- 审查员 → 可访问 /projects 及其子页面
- 管理员 → 可访问 /projects（查看所有项目）+ /admin 下所有页面

**验收标准 (AC)**:
1. 未登录用户访问任何非 /login 页面，重定向至 /login
2. 审查员访问 /admin/* 页面，显示403或重定向至 /projects
3. 管理员可访问所有页面
4. JWT 过期后自动重定向至登录页，提示"登录已过期，请重新登录"
5. **表单防丢失**: 前端 axios 拦截器收到 401 时，若当前页面有未提交的表单数据（如复核意见），先将表单内容缓存到 localStorage，登录成功后自动恢复并提示用户确认提交
6. 后端每个 API 端点都验证 JWT 有效性和角色权限

**测试验证点**:
- **后端 API 测试**:
  - 无 token 请求 → 401
  - 过期 token → 401
  - 审查员 token 访问 admin API → 403
  - 管理员 token 访问 admin API → 200
- **前端 E2E 测试**:
  - 审查员登录 → 导航栏无管理菜单 → 手动输入 /admin/users → 重定向至 /projects
  - Token 过期 → 页面显示过期提示 → 跳转登录页

### US-1.4 初始用户与修改密码

**作为** 系统管理员，
**我希望** 系统部署后有默认管理员账号，首次登录后强制修改密码，
**以便** 安全地完成系统初始化。

**描述**:
- 数据库初始化脚本自动创建默认管理员（admin / admin123），标记 must_change_password=true
- 默认管理员首次登录后，前端拦截跳转到修改密码页面，修改前不允许执行其他操作
- 所有用户可通过个人设置修改自己的密码

**验收标准 (AC)**:
1. 数据库初始化后存在 admin 用户，can login
2. 首次登录 `GET /api/auth/me` 返回 must_change_password=true
3. must_change_password=true 时，前端强制跳转修改密码页，屏蔽其他操作
4. `POST /api/auth/change-password` {old_password, new_password} → 修改成功后 must_change_password=false
5. 新密码规则：≥8位，包含字母和数字
6. 旧密码错误 → 400

**测试验证点**:
- **后端 API 测试**:
  - 初始化后 → admin 用户存在，密码为 admin123 的 bcrypt 哈希
  - `POST /api/auth/change-password` 正确旧密码 + 合规新密码 → 200
  - `POST /api/auth/change-password` 错误旧密码 → 400
  - `POST /api/auth/change-password` 新密码不满足规则 → 422
- **前端 E2E 测试**:
  - admin 首次登录 → 自动跳转修改密码页
  - 修改密码后 → 可正常使用系统

## Epic 2: 项目管理

### US-2.1 创建检测项目

**作为** 审查员，
**我希望** 创建一个新的检测项目并填写基本信息，
**以便** 开始一次围标检测工作。

**`[GAP]` 前后端对齐**:
- 后端: `POST /api/projects/` 当前为空壳 TODO，需实现完整逻辑
- 后端: 需创建 Project 数据库模型（name, bid_code, max_price, description, status, owner_id, created_at）
- 后端: 需创建 ProjectCreate / ProjectResponse Pydantic schema
- 前端: 需创建 /projects/new 页面

**描述**:
审查员点击"新建项目"按钮，进入创建项目表单。填写以下字段：
- 项目名称（必填，≤100字符）
- 招标编号（选填，≤50字符）
- 最高限价（选填，数字，用于"报价接近限价"维度检测）
- 项目描述（选填，≤500字符）

创建成功后跳转至项目详情页。

**验收标准 (AC)**:
1. 项目名称为必填，为空时表单不可提交并显示校验提示
2. 最高限价仅接受正数，小数点后最多2位
3. 创建成功后项目状态为"待上传(draft)"，跳转至项目详情页
4. 项目自动关联当前登录用户为 owner
5. 同一用户可创建多个同名项目（不做唯一性校验）

**`[RISK]` 风险点**: 
最高限价为选填，但如果不填，"报价接近限价"维度将自动跳过。需在创建页面给予文案提示。

**测试验证点**:
- **后端 API 测试**:
  - `POST /api/projects/` 合法数据 → 201 + 返回完整项目对象（含 id, status="draft"即"待上传"）
  - `POST /api/projects/` 缺少 name → 422 验证错误
  - `POST /api/projects/` max_price 为负数 → 422
  - 不同用户创建的项目互相隔离（审查员只看到自己的）
- **前端 E2E 测试**:
  - 不填项目名称点提交 → 显示"项目名称不能为空"
  - 填写完整信息提交 → 跳转至项目详情页
  - 不填最高限价 → 显示提示文案"未设置最高限价，将跳过'报价接近限价'维度检测"

### US-2.2 项目列表

**作为** 审查员，
**我希望** 查看我创建的所有检测项目，
**以便** 快速找到并进入目标项目。

**`[GAP]` 前后端对齐**:
- 后端: `GET /api/projects/` 当前返回空数组，需实现分页、筛选、权限过滤
- 前端: 需创建 /projects 页面，安装状态管理

**描述**:
项目列表以卡片网格展示，每张卡片包含：项目名称、招标编号、状态、风险等级（检测完成后显示）、创建时间。

支持功能：
- 按状态筛选（全部/草稿/解析中/检测中/已完成）
- 按风险等级筛选（全部/高风险/中风险/低风险）
- 关键词搜索（匹配项目名称、招标编号）
- 按创建时间倒序排列

**验收标准 (AC)**:
1. 审查员仅看到自己创建的项目
2. 管理员可看到所有用户的项目
3. 卡片上风险等级用颜色区分：红色(高)、橙色(中)、绿色(低)、灰色(未检测)
4. 空状态时显示"暂无项目，点击新建"引导按钮
5. 支持分页，每页12个项目

**测试验证点**:
- **后端 API 测试**:
  - `GET /api/projects/` 审查员 A → 只返回 A 创建的项目
  - `GET /api/projects/` 管理员 → 返回所有项目
  - `GET /api/projects/?status=draft` → 只返回草稿状态
  - `GET /api/projects/?risk_level=high` → 只返回高风险
  - `GET /api/projects/?search=xxx` → 名称/编号模糊匹配
  - `GET /api/projects/?page=1&size=12` → 分页正确
- **前端 E2E 测试**:
  - 无项目时 → 显示空状态引导
  - 有项目时 → 卡片正确展示各字段
  - 点击筛选 → 列表实时更新
  - 点击卡片 → 跳转至项目详情页

### US-2.3 项目详情（操作中枢）

**作为** 审查员，
**我希望** 在项目详情页管理投标人、文件、启动检测、查看结果，
**以便** 在一个页面内完成整个检测流程。

**描述**:
项目详情页是核心操作页面，包含以下区域：
1. **项目基本信息** — 名称、编号、限价、状态（顶部）
2. **投标人管理区** — 投标人列表、添加/删除投标人
3. **文件管理区** — 每个投标人下的文件列表、上传入口、解析状态
4. **操作区** — "启动检测"按钮（≥2个投标人且解析完成时可点击）
5. **检测进度区** — 检测中时显示各 Agent 状态（SSE驱动）
6. **结果入口** — 检测完成后显示"查看报告"链接

**验收标准 (AC)**:
1. 页面顶部显示项目状态流转：草稿(draft) → 解析中(parsing) → 待检测(ready) → 检测中(analyzing) → 已完成(completed)
2. 投标人不足2个时，"启动检测"按钮禁用并显示提示
3. 存在未解析完成的文件时，"启动检测"按钮禁用
4. 检测中时，页面不可删除投标人/文件
5. 检测完成后显示风险等级徽章和"查看报告"入口
6. 已完成项目可追加投标人，重新启动全量检测

**测试验证点**:
- **后端 API 测试**:
  - `GET /api/projects/{id}` → 返回项目详情 + 投标人列表 + 文件列表 + 当前状态
  - 非 owner 的审查员访问 → 404（数据隔离）
- **前端 E2E 测试**:
  - 项目状态=draft → 显示"添加投标人"引导
  - 项目状态=analyzing → "启动检测"按钮变为"检测中..."，不可点击
  - 项目状态=completed → 显示风险等级和报告入口

### US-2.4 删除项目

**作为** 审查员，
**我希望** 删除不再需要的项目，
**以便** 保持项目列表整洁。

**描述**:
审查员在项目列表或详情页点击"删除"，弹出二次确认对话框。确认后**软删除**该项目（标记 `deleted_at`，不再出现在列表/详情中），DB 记录保留以便审计。所有关联数据（投标人、文件、解析产物、分析结果、报告）随项目软删一并隐藏；其磁盘文件由数据生命周期任务（C1 骨架 + C4 投标人/文件实体落地后接管）统一清理。

**验收标准 (AC)**:
1. 删除前弹出确认对话框，显示项目名称
2. 检测中（status=analyzing）的项目不可删除，提示"检测进行中，无法删除"
3. 软删除：将 `deleted_at` 置为当前时间；列表/详情端点默认过滤 `deleted_at IS NULL`，对调用方表现为"项目已消失"；DB 记录保留供审计与潜在恢复
4. 非 owner 的 reviewer 调用 DELETE 返回 **404**（不返 403），避免泄露项目存在性
5. 删除后返回项目列表页
6. 关联文件清理：交由数据生命周期清理任务在 C4 文件实体落地后接管；C3 阶段项目无物理文件挂载，无需在删除路径内同步清盘

**`[Change History]`**:
- 2026-04-14（C3 project-mgmt 归档）：将原 AC-3 由"硬删除并清盘"修订为"软删除（`deleted_at`）+ 文件清理移交生命周期任务"，与 `docs/execution-plan.md` §C3 验证场景 4 对齐；新增 AC-4 显式约束"非 owner 返 404 防存在性泄露"

**测试验证点**:
- **后端 API 测试**（C3 已实现于 `backend/tests/e2e/test_projects_api.py`）：
  - `DELETE /api/projects/{id}` status=draft → 204，DB 中 `deleted_at` 已置值，列表中不再出现
  - `DELETE /api/projects/{id}` status=analyzing → 409 冲突
  - `GET /api/projects/` → 已软删项目不出现
  - reviewer 删他人项目 → 404（不返 403）
  - 已软删项目重复删除 → 404
- **前端 E2E 测试**（C3 已实现于 `e2e/tests/c3-project-crud.spec.ts`）：
  - 点击删除 → 弹出确认框 → 取消 → 项目仍在
  - 确认删除 → 项目从列表消失

## Epic 3: 投标人与文件管理

### US-3.1 添加投标人并上传文件包

**作为** 审查员，
**我希望** 在弹窗中输入投标人名称并同时上传其文件压缩包，
**以便** 一步完成投标人创建和文件上传。

**`[GAP]` 前后端对齐**:
- 后端: 无投标人相关 API，需新增：
  - `POST /api/projects/{project_id}/bidders` — 创建投标人（含文件上传）
  - `GET /api/projects/{project_id}/bidders` — 列表
  - `DELETE /api/projects/{project_id}/bidders/{bidder_id}` — 删除
- 后端: 需创建 Bidder 模型（name, project_id, identity_info, total_bid_price, parse_status, file_count）

**描述**:
在项目详情页点击"添加投标人"按钮，弹出弹窗：
- 输入投标人名称（公司名）
- 拖拽/选择压缩包上传区域（ZIP/RAR/7Z）
- 提交后同时创建投标人记录并触发文件解析

此交互将添加投标人和上传文件合并为一步操作，符合设计文档"操作极简化"原则。

**验收标准 (AC)**:
1. 投标人名称必填，≤200字符
2. 同一项目内投标人名称不可重复
3. 压缩包为选填——允许先创建投标人后再上传（但建议同时上传）
4. 支持 ZIP/RAR/7Z 格式，其他格式前端拦截提示
5. 文件大小超过500MB前端拦截提示
6. 提交后投标人出现在列表中，若已上传文件则状态为"解析中"，未上传则为"待上传"
7. 投标人身份信息（公司全称、简称、关键人员、资质编号等）由后续 LLM 解析自动提取

**测试验证点**:
- **后端 API 测试**:
  - `POST /api/projects/{pid}/bidders` {name: "A公司", file: ZIP} → 201
  - `POST /api/projects/{pid}/bidders` {name: "A公司"} 无文件 → 201，status=pending
  - 同项目重复名称 → 409
  - 空名称 → 422
  - 上传 .exe 文件 → 415
  - 超过500MB → 413
- **前端 E2E 测试**:
  - 弹窗输入名称+拖拽ZIP → 提交 → 投标人出现在列表，状态"解析中"
  - 弹窗只输入名称不上传文件 → 提交 → 投标人出现在列表，状态"待上传"
  - 拖拽 PDF 文件 → 提示"不支持的文件格式"

### US-3.2 追加上传文件包

**作为** 审查员，
**我希望** 为已有投标人追加上传额外的压缩包，
**以便** 补充该投标人的投标文件。

**`[GAP]` 前后端对齐**:
- 后端: `POST /api/projects/{pid}/bidders/{bid}/upload` 当前只返回文件名，需完整实现：
  - 接收 file 参数（bidder 由路径参数指定）
  - 文件保存到 uploads 目录（隔离存储）
  - 创建 BidDocument 记录
  - 触发异步解压+解析任务
  - **追加模式**：新文件与已有文件合并，不替换
- 后端: 异步解压+解析通过 asyncio.create_task + run_in_executor(ProcessPoolExecutor) 实现，不使用 Celery/Redis
- 前端: 需实现拖拽上传组件、进度条

**`[RISK]` 风险点**:
- 压缩包可能包含恶意文件（zip bomb）→ 需在解压时限制：总大小≤2GB、文件数≤1000、嵌套层数≤3
- 单文件最大100MB，单个压缩包最大500MB
- 追加上传可能产生重复文件 → 需基于 MD5 去重并提示用户

**描述**:

## Epic 4: 文档解析

### US-4.1 压缩包解压与安全校验

**作为** 系统，
**我希望** 安全地解压用户上传的压缩包，
**以便** 提取其中的投标文件用于后续分析。

**`[GAP]` 前后端对齐**:
- 后端: parser 服务目前为空模块，需完整实现
- 后端: 异步解压通过 asyncio.create_task + run_in_executor(ProcessPoolExecutor) 实现

**`[RISK]` 风险点**:
- Zip bomb 攻击：恶意压缩包解压后可能占满磁盘 → 必须在解压过程中实时监控
- **Zip Slip 目录穿越**：恶意压缩包内含 `../../` 路径可覆写服务器任意文件 → 解压前校验每个条目路径，拒绝 `..` 和绝对路径
- 嵌套压缩包：压缩包内包含压缩包，需递归解压但限制深度
- 文件名编码：中文文件名在 ZIP 中可能使用 GBK 编码 → 需自动检测编码

**描述**:
用户上传压缩包后，后台异步任务执行解压：
1. 校验压缩包完整性
2. 递归解压（最多3层嵌套）
3. 解压过程中实时检查：总大小≤2GB、文件总数≤1000
4. 处理文件名编码问题（UTF-8 / GBK / GB2312 自动检测）
5. 解压完成后，按文件类型分类登记到 BidDocument 表

**验收标准 (AC)**:
1. ZIP/RAR/7Z 格式均可正常解压
2. 解压总大小超过2GB时终止并标记错误"解压文件过大，超过2GB限制"
3. 文件总数超过1000时终止并标记错误
4. 嵌套超过3层的压缩包终止并标记
5. GBK 编码的中文文件名正确还原
6. **Zip Slip 防护**：含 `../` 路径的条目被跳过并记录警告，不解压到目标目录外
7. 损坏的压缩包标记错误"文件已损坏，无法解压"
7. **加密压缩包**标记为"需密码"状态，前端提示用户输入密码后重试解压
8. 单个文件超过100MB时标记警告，尝试分块处理，设超时保护
9. 解压进度通过 SSE 推送到前端

**测试验证点**:
- **后端 API 测试**:
  - 正常 ZIP → 解压成功，BidDocument 记录创建
  - 嵌套2层 ZIP → 递归解压成功
  - 嵌套4层 ZIP → 到第3层停止，标记警告
  - 含 `../../etc/passwd` 路径的恶意 ZIP → 该条目跳过，其他文件正常解压，日志记录警告
  - 超大压缩包（解压>2GB）→ 中止，错误状态
  - GBK 编码文件名 → 正确解码
  - 损坏的 ZIP → 错误状态 + 原因
  - 空压缩包 → 错误"压缩包内无有效文件"
  - 加密 ZIP → 标记"需密码"，`POST /api/documents/{id}/decrypt` {password} → 重新解压
  - 单文件>100MB → 标记警告，设超时保护
- **前端 E2E 测试**:
  - 加密压缩包 → 投标人状态显示"需密码" → 点击"输入密码"→ 弹窗输入 → 重新解压

### US-4.2 文档内容提取

**作为** 系统，
**我希望** 从解压后的文件中提取文本、元数据和图片信息，
**以便** 为后续检测分析提供结构化数据。

**描述**:
对解压后的每个文件按类型分别处理：

**DOCX 文件**:
- 提取全文文本，保留段落结构（存入 DocumentText）
- 提取元数据：作者、最后修改人、公司名、创建时间、修改时间、创建软件（存入 DocumentMetadata）
- 提取文本框、页眉页脚中的文本
- 提取嵌入图片，计算 MD5 + pHash（存入 DocumentImage）

**XLSX 文件**:
- 提取所有 sheet 的原始单元格数据
- 提取元数据（同 DOCX）
- 提取隐藏 sheet/隐藏列中的信息（可能包含硬件指纹）
- 注: 报价表结构识别和报价数据提取在流水线步骤4中执行（见 requirements.md F-PA-05），不在本步骤

**图片文件** (JPG/PNG/BMP/TIFF):
- 计算 MD5 哈希
- 计算感知哈希 pHash
- 记录文件大小、位置（存入 DocumentImage）

**`[RISK]` 风险点**:
- XLSX 报价表结构千差万别，LLM 首次识别结构后，该结构规则应用到同项目所有投标人 → 需要 PriceParsingRule 表存储规则
- LLM 识别失败时的兜底：标记"无法识别报价表结构"，该维度跳过

**验收标准 (AC)**:
1. DOCX 段落完整提取，短段落（<50字符）标记但不过滤（保留原始数据，相似度计算时过滤，阈值管理员可配置）
2. DOCX 文本框内容提取，归入所在位置的文本流中
3. DOCX **页眉页脚单独提取**，用于错误一致性检测（公司名混淆），**不参与文本相似度计算**
4. DOCX 表格内文本按行合并为一个比对单元
5. DOCX 元数据 author/last_saved_by/company 正确提取
4. XLSX 通过 LLM 正确识别报价表头和列映射
5. 图片 MD5 和 pHash 正确计算
6. 不支持的文件格式（DOC/XLS/PDF）标记为 skipped，不报错
7. 单个文件解析失败不影响其他文件
8. 所有提取结果存入对应数据库表

**测试验证点**:
- **后端 API 测试**:
  - 标准 DOCX → DocumentText.paragraphs 非空，段落数 > 0
  - 含文本框的 DOCX → 文本框内容也在提取结果中
  - 标准 XLSX 报价表 → PriceItem 记录正确（item_name, unit_price, quantity 等）
  - XLSX 结构异常 → LLM 返回识别结果或标记失败
  - JPG 图片 → DocumentImage.image_md5 为32位十六进制，image_phash 为64位二进制
  - .doc 文件 → parse_status="skipped"
  - 单个 DOCX 解析异常 → 其他文件继续解析

### US-4.3 文档角色分类与投标人标识提取

**作为** 系统，
**我希望** 通过 LLM 对每个投标人的文档进行角色分类，并同时提取投标人标识信息，
**以便** 在检测时对比同角色文档，并利用标识信息进行错误一致性检测。

**`[GAP]` 前后端对齐**:
- 后端: 需在 BidDocument 模型中添加 doc_role 字段和 role_confidence 字段
- 后端: 需实现 LLM 调用接口（统一 LLM 适配层）
- 后端: 第一期 Prompt 在代码中维护（Python 模块或 YAML），不建 PromptTemplate 表
- 后端: 需更新 Bidder 模型的 identity_info 字段

**描述**:
解析完成后，系统按投标人批量处理：将一个投标人的全部文件名+每个文件的首段文本发送给 LLM，**一次调用同时完成两项任务**：

**任务1: 文档角色分类**
分类策略为规则预分类（文件名关键词匹配）+ LLM 批量确认/修正。文档角色定义（共9种）：

| 角色 | 识别依据 |
|------|---------|
| 技术方案 (technical) | 技术方案、技术标、技术建议书 |
| 施工组织设计 (construction) | 施工组织、施工方案、施工设计 |
| 报价清单 (pricing) | 报价、清单、工程量、商务标、投标报价 |
| 综合单价分析表 (unit_price) | 综合单价、单价分析 |
| 投标函 (bid_letter) | 投标函、投标书 |
| 资质证明 (qualification) | 资质、资格、营业执照 |
| 企业介绍 (company_intro) | 企业介绍、公司简介、公司概况 |
| 授权委托书 (authorization) | 授权、委托 |
| 其他 (other) | 以上均不匹配 |

**任务2: 投标人标识信息提取**
从文件内容中提取该投标人的标识信息，存入 Bidder.identity_info：
- 公司全称、简称
- 关键人员姓名（项目经理、法人代表等）
- 资质编号
- 联系方式

这些标识信息用于后续**错误一致性**维度的交叉检索。

**`[RISK]` 风险点**:
- LLM 分类错误会导致错误的文档对比 → 用户应可在文件列表中手动修改文档角色
- LLM 服务不可用时的兜底：使用文件名关键词规则匹配（管理员可配置关键词列表）
- LLM 置信度低的分类应标记为"待确认"，前端高亮提示引导用户修正

**验收标准 (AC)**:
1. 每个 DOCX/XLSX 文件都有角色标签（9种角色之一）
2. LLM 不可用时，回退到关键词规则匹配
3. LLM 分类置信度低的文件标记为"待确认"，前端以黄色高亮标签显示
4. 用户可在文件列表中点击角色标签弹出下拉修改
5. 角色修改后不需要重新解析，不自动改变项目状态。但若项目 status=completed，前端在详情页顶部展示提示 banner："文档角色已修改，当前报告基于修改前的分类结果，建议重新检测"
6. 投标人标识信息（identity_info）正确提取并存入 Bidder 记录
7. 角色识别关键词由管理员在规则配置中维护

**测试验证点**:
- **后端 API 测试**:
  - 提供投标人全部文件名+首段 → LLM 返回 {roles: [...], identity_info: {...}}
  - LLM 超时 → 回退关键词匹配，文件名含"报价" → "pricing"
  - LLM 返回低置信度 → role_confidence="low"，前端标记"待确认"
  - `PATCH /api/documents/{id}/role` {role: "technical"} → 200
  - 角色分类后 Bidder.identity_info 包含公司全称/简称/关键人员
- **前端 E2E 测试**:
  - 解析完成后 → 文件列表每个文件显示角色标签
  - 低置信度文件 → 黄色高亮"待确认"标签
  - 点击角色标签 → 弹出下拉可修改

### US-4.4 报价解析规则确认与修正

**作为** 审查员，
**我希望** 查看并确认/修正 LLM 识别的报价表解析规则，
**以便** 确保报价数据提取的准确性。

**`[GAP]` 前后端对齐**:
- 后端: 需新增 `GET /api/projects/{pid}/price-rules` — 获取 LLM 识别的解析规则
- 后端: 需新增 `PUT /api/projects/{pid}/price-rules/{id}` — 修正规则
- 后端: PriceParsingRule 模型已在设计文档中定义

**描述**:
LLM 识别报价表结构后（表头行、列映射关系），系统在项目详情页展示识别结果供审查员确认。审查员可修正列映射（如"哪列是编码、哪列是单价"）。确认/修正后，系统用该规则批量应用到所有投标人的报价表。

同一项目内 LLM 只需理解一次格式（招标方通常提供统一模板），输出解析规则后程序批量应用。

**验收标准 (AC)**:
1. LLM 识别完成后，在项目详情页显示"报价解析规则待确认"提示
2. 展示识别到的 sheet 名称、表头行号、列映射关系
3. 审查员可修改列映射（下拉选择：项目编码/项目名称/单位/数量/单价/合价/跳过）
4. 规则生成/确认后自动批量应用到所有投标人（含已解析完成但尚未提取报价的投标人，自动回填）
5. 同一项目多个投标人同时解析时，报价结构识别仅执行一次（并发控制），后续投标人等待规则后直接应用
6. 批量应用失败的投标人，单独调用 LLM 重新理解表结构；仍失败则标记"报价解析失败"
7. 已由 LLM 生成的规则标记 created_by_llm=true，用户修正后标记为 false

**测试验证点**:
- **后端 API 测试**:
  - `GET /api/projects/{pid}/price-rules` → 返回 LLM 识别的规则列表

### US-5.2 Agent 并行执行与自检

**作为** 系统，
**我希望** 各检测维度 Agent 独立并行运行，各自检查前置条件，
**以便** 最大化并行度，某个 Agent 失败不阻塞其他 Agent。

**描述**:
每个 Agent 启动时先自检前置条件：

| Agent | 前置条件 | 不满足时 |
|-------|---------|---------|
| 硬件指纹 | 提取到硬件信息 | 跳过："未检测到定价软件输出" |
| 报价结构 | 存在报价单文件 | 跳过："未找到报价表" |
| 报价规律 | ≥3个投标人有报价 | 跳过："有效投标人不足" |
| 限价接近 | 项目设置了最高限价 | 跳过："未设置最高限价" |
| 文本相似度 | 同角色文档存在 | 跳过："缺少可对比文档" |
| 错误一致性 | 提取到身份信息 | **降级运行**：identity_info 为空时，退化为用投标人名称（Bidder.name）做纯关键词交叉搜索，不跳过。降级模式下不做铁证判定，仅标记"降级检测，建议补充标识信息后重新检测" |
| 语言风格 | 同角色文档存在 | 跳过，≥20投标人自动分组 |
| 图片复用 | 提取到图片 | 跳过："未提取到图片" |
| 软件元数据 | 提取到元数据 | 跳过："未提取到元数据" |
| 操作时间 | ≥3个投标人有元数据 | 跳过："有效投标人不足" |

**验收标准 (AC)**:
1. 各 Agent 作为独立的 asyncio 协程通过 asyncio.gather 并行运行（CPU 密集部分通过 run_in_executor 卸载到进程池）
2. 单个 Agent 失败（异常/超时）不影响其他 Agent 运行
3. Agent 自检不满足条件时，状态标记为"skipped"并记录原因
4. Agent 超时限制：单个 Agent 最大运行时间 5 分钟
5. 所有 Agent 完成/跳过/失败后，触发 LLM 综合结论生成
6. Agent 结果写入 PairComparison 或 OverallAnalysis 表

**`[RISK]` 风险点**:
- LLM 调用频繁：10个维度中有6个需要 LLM → 需要重试机制（第一期不做速率限制，被限流时走重试兜底）
- LLM 返回格式不符预期 → JSON 解析失败的兜底（重试1次，仍失败则标记agent失败，使用程序得分）
- 所有 Agent 全部跳过的极端情况 → 报告结论应为"无法进行有效检测，数据不足"

**测试验证点**:
- **后端 API 测试**:
  - 启动检测 → 所有 Agent task 创建成功
  - 无报价表的投标人 → 报价相关 Agent 状态 = "skipped"
  - identity_info 为空的投标人 → 错误一致性 Agent 降级运行（用投标人名称搜索），不跳过
  - 投标人简称为高频词（如"工程"）→ 防碰撞过滤生效，LLM 候选段落数≤100
  - Agent 内部异常 → 状态 = "failed"，其他 Agent 不受影响
  - Agent 超时 → 状态 = "timeout"
  - 所有 Agent 完成后 → 触发综合结论

### US-5.4 检测超时与全局兜底

**作为** 系统，
**我希望** 检测过程有全局超时控制和异常兜底，
**以便** 不会因为单个环节卡住导致整个检测永远无法完成。

**`[RISK]` 风险点**: 此 Story 是系统稳定性的关键保障

**描述**:
- 单个 Agent 超时：5分钟强制终止，标记 timeout
- 全局检测超时：30分钟后强制结束所有未完成 Agent
- LLM 调用超时：单次30秒，**重试2次**
- LLM 调用失败：该维度仅展示程序化结果，标记"AI 研判暂不可用"
- 综合结论 LLM 调用失败：仅展示各维度的程序化评分和数据，不生成自然语言结论
- 报价解析规则批量应用失败：对失败的投标人单独调用 LLM 重新理解表结构；仍失败则标记"报价解析失败"

**验收标准 (AC)**:
1. Agent 超时后状态标记为"timeout"，记录已运行时长
2. 全局超时后，未完成 Agent 全部标记为"timeout"，触发综合结论
3. LLM 单次调用超时后重试2次，仍失败则 Agent 使用程序得分，标记"AI 研判暂不可用"
4. 综合结论 LLM 失败时，仅展示程序化评分，不生成结论文本
5. 报价解析批量应用失败的投标人 → 单独 LLM 重试 → 仍失败则价格相关维度跳过
6. 所有超时和兜底情况在检测日志中有详细记录

**测试验证点**:
- **后端 API 测试**:
  - 模拟 Agent 运行超5分钟 → 强制终止，status="timeout"
  - 模拟 LLM 接口不可用 → Agent 使用程序得分完成
  - 模拟综合结论 LLM 失败 → 模板化结论生成
  - 超时 Agent 不阻塞报告生成

## Epic 12: 技术 Story（补充脱节点与风险点）

> 以下 Story 不直接面向用户操作，但属于系统正常运行的必要基础设施。
> 在前后端分析中发现的脱节和风险点集中在此补充。

### TS-1 异步任务基础设施

**描述**:
第一期采用 asyncio + ProcessPoolExecutor 实现异步任务，不引入 Celery/Redis：
- CPU 密集任务（文档解压、哈希计算、TF-IDF）通过 `run_in_executor(ProcessPoolExecutor)` 卸载到子进程
- IO 密集任务（LLM API 调用）直接使用 asyncio 协程
- 后台任务通过 `asyncio.create_task` 启动，不阻塞 API 响应
- 任务状态变化更新数据库 + 推送 SSE 事件
- **进程级超时终止**: `asyncio.Task.cancel()` 无法终止 ProcessPoolExecutor 中已运行的子进程。超时后必须通过 `Process.kill()` 强制终止卡死的 worker 并回收资源，防止僵尸进程累积

**已知限制**: 进程重启后未完成任务丢失，需用户手动重新触发。

**验收标准 (AC)**:
1. 上传压缩包后，解析任务在后台执行，API 立即返回
2. 检测启动后，各 Agent 通过 asyncio.gather 并行运行
3. 任务状态变化实时更新数据库
4. 任务完成后触发 SSE 推送
5. 单个任务超时（5 分钟）后自动标记为 timeout
6. **进程重启恢复**: 系统启动时扫描并修复异常状态——parsing 状态的 BidDocument 重置为 failed，parsing 状态的 Bidder 根据文件状态重新计算，analyzing 状态的 Project 重置为 ready，running 状态的 AgentTask 标记为 failed（详见 requirements.md 3.2 状态机）

**测试验证点**:
- 提交解析任务 → 后台执行 → 数据库状态更新正确
- CPU 密集任务 → 不阻塞 FastAPI 事件循环（其他 API 仍响应正常）
- 任务超时 → 状态标记为 timeout，对应的 ProcessPoolExecutor worker 进程被 kill 回收（无残留僵尸进程）
- 模拟进程重启 → 异常状态的记录被正确重置 → 用户可重新触发

---

### TS-2 SSE 推送基础设施

**描述**:
使用 FastAPI 内置 `EventSourceResponse`（v0.135+）实现最简 SSE 推送：
- SSE 端点 `GET /api/projects/{pid}/events`，返回 async generator
- 事件类型：`parse_progress`, `agent_status`, `report_ready`
- 权限校验：只能订阅自己项目的事件（非 owner 返回 403）
- **不做**自定义心跳管理、连接池、断线补发。浏览器 EventSource 自带自动重连（默认 3 秒）；重连后前端调 API 拉取最新状态恢复

**验收标准 (AC)**:
1. SSE 端点返回 content-type: text/event-stream
2. 任务状态变化时推送对应事件
3. 多个客户端可同时订阅同一项目
4. 非 owner 订阅返回 403
5. 前端 SSE 断线重连后通过 `GET /api/projects/{pid}/analysis/status` 恢复完整 Agent 级状态（含 score/summary），不依赖仅有汇总计数的项目详情接口

**测试验证点**:
- SSE 连接建立 → 任务状态变化时收到事件
- 非 owner 连接 → 403

### TS-3 LLM 统一适配层

**`[GAP]` 关键脱节**: 设计文档要求支持多种 LLM（Claude/GPT/Qwen/DeepSeek），但后端无任何 LLM 调用代码。

- 统一接口：`llm.chat(prompt, model_config) -> LLMResponse`
- 支持的协议：OpenAI 兼容 API（覆盖大多数模型）
- 重试机制：超时30秒，重试2次
- 结构化输出解析：JSON 格式校验和容错

### TS-6 数据生命周期管理

**`[RISK]` 风险点**: 原始文件90天后自动清理，但分析结果永久保留。需要定时任务支持。

**描述**:
实现文件自动清理机制：
- 定时任务（系统 cron 或 FastAPI 启动时注册的定时协程）每日执行
- 清理超过 N 天（默认90，可配置）的：压缩包原文件、解压后的文档文件、图片文件
- 保留数据库中的分析结果（DocumentText, PriceItem, DocumentImage 的哈希值等）
- 清理前记录日志

**验收标准 (AC)**:
1. 清理任务每日自动执行
2. 只清理超过配置天数的文件
3. 数据库记录不受影响
4. 清理后用户仍可查看报告和对比视图（基于已存储的结构化数据）
5. 清理后用户无法下载原始文件，显示"原始文件已清理"
6. 清理日志可查

**测试验证点**:
- 创建91天前的文件 → 执行清理 → 文件删除，数据库记录在
- 创建89天前的文件 → 执行清理 → 文件保留
- 清理后访问报告 → 正常显示
- 清理后下载原始文件 → 提示"已清理"
