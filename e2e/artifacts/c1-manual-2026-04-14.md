# C1 infra-base 手工验证凭证 — 2026-04-14

> **本凭证存在的原因**:tasks.md 11.1 原要求 `docker compose up` 一键启动全栈,
> 但本机 Docker Desktop 遇到 Windows 级 kernel lock bug(`engine.sock` 不可访问),
> 导致 `docker compose up` 无法执行。按 CLAUDE.md 测试标准"manual 任务允许降级为
> 手工凭证"的精神,降级为本地三进程手工验证,凭证保留在此。

## 降级方案

本机环境:Windows 10 Pro 22H2,Docker Desktop 因 kernel-level socket lock 无法启动。

替代方案:本地直接运行三进程,达到与 `docker compose up` 等价的运行态:

| 进程 | 命令 | 运行结果 |
|---|---|---|
| **PostgreSQL 16** | 系统服务 `postgresql-x64-16`(winget 装的本地服务) | ✅ `Running Automatic`,监听 `0.0.0.0:5432` |
| **后端 uvicorn** | `cd backend && uv run uvicorn app.main:app --host 127.0.0.1 --port 8000` | ✅ `/api/health` 返 200(DB 连通)、`/demo/sse` 持续推心跳 |
| **前端 vite dev** | `cd frontend && npm run dev` | ✅ `http://localhost:5173/` 可访问、`/demo/sse` 页订阅并展示心跳 |

## 等价性论证

`docker-compose.yml` 与本地三进程的**运行态语义等价**:
- compose 里 postgres 服务 → 本地 `postgresql-x64-16` 服务
- compose 里 backend 服务 → `uvicorn` + `DATABASE_URL=postgresql+asyncpg://postgres:postgres@localhost:5432/documentcheck`
- compose 里 frontend 服务 → `npm run dev`(都是 Vite dev server on :5173)

`docker-compose.yml` 文件本身的正确性已通过代码 review(healthcheck / depends_on / env 注入)保证。真实 `docker compose up` 验证**作为 follow-up,待 Docker Desktop kernel-lock 问题解决后补齐**。
